# Aircraft Assets And Attribution

A3S Agent Evaluation currently bundles procedural Three.js aircraft geometry generated by
the application. No third-party mesh, texture, logo, or downloaded Sketchfab
binary is included in the repository.

Each built-in aircraft is an independent blueprint assembled from reusable
fuselage, surface, propulsion, cockpit, livery, pilot, and loadout modules. The
generic prototype uses the same assembly path and remains the fallback for an
unrecognized model or Candidate.

## Built-in airframe catalog

| Airframe | Distinguishing procedural geometry | Default use | Bundled source |
| --- | --- | --- | --- |
| J-50 | Tailless lambda planform, twin engines, low canopy | A3S Code / GLM-5.2 | A3S procedural generator |
| J-35 | Compact trapezoid wing, twin round exhausts, canted twin tails | Optional carrier-multirole specimen | A3S procedural generator |
| F-35 Lightning II | Short, deep single-engine body and serrated round exhaust | Codex | A3S procedural generator |
| F-22 Raptor | Wide diamond wing, large stabilizers, twin rectangular exhausts | Claude Code | A3S procedural generator |
| Generic Test Prototype | Forward-swept experimental planform and neutral markings | Unknown or custom Candidate | A3S procedural generator |

The default shared-flow formation uses stable Agent-to-airframe identities:

| Agent | Airframe |
| --- | --- |
| A3S Code (`anthropic/glm-5.2`) | J-50 |
| Codex | F-35 Lightning II |
| Claude Code | F-22 Raptor |

These defaults do not prevent composition. A hangar roster can explicitly pair
any pilot with any of the five built-in airframes, and that explicit selection
takes priority over model-family fallback resolution. A separate pilot registry
assigns A3S blue/indigo spectrum attire, Codex graphite/green orbit attire,
Claude warm gray/orange ray attire, or a violet generic test suit. A third
registry maps visualization effort to procedural air-to-air stores.

The markings are product-level identity treatments, not official aircraft
liveries or reproductions of third-party logos.

## Adding an airframe

New procedural airframes do not require changes to the generic assembler.

1. Add an ID to `BuiltInAircraftId` in `aircraft-blueprint.ts`.
2. Add one entry to `AIRCRAFT_BLUEPRINTS` in `aircraft-blueprints.ts` by starting
   from the generic prototype dimensions.
3. Define the fuselage loft, wing and tail planforms, engine offsets, intake
   spacing, and cockpit dimensions. `defineAircraftBlueprint()` derives engine
   and vertical-tail counts and rejects inconsistent tailless layouts.
4. Add the airframe to `HANGAR_AIRFRAME_OPTIONS` so it can be composed explicitly.
5. Add public display metadata to `AIRFRAMES`. Add a model-family rule only when
   the airframe should resolve automatically outside an explicit roster.
6. Add or update a fixed Agent mapping only when the product identity requires
   the new airframe in the default formation.
7. Add blueprint structure, visual-distinction, rendered bounding-box, hangar
   catalog, and formation-mapping tests.

For experiments or adapters, any string ID can be assembled without modifying
the built-in registry:

```ts
import { assembleAircraft } from '../src/components/scene/aircraft-assembler';
import { defineAircraftBlueprint } from '../src/components/scene/aircraft-blueprint';
import { AIRCRAFT_BLUEPRINTS } from '../src/components/scene/aircraft-blueprints';

const prototype = AIRCRAFT_BLUEPRINTS.prototype;
const demonstrator = defineAircraftBlueprint({
  ...prototype,
  id: 'adapter-demonstrator',
  displayName: 'Adapter Demonstrator',
  layout: {
    wing: prototype.layout.wing,
    tailless: prototype.layout.tailless,
  },
});

const aircraft = assembleAircraft(demonstrator, {
  accentColor: '#69d6c5',
  livery: 'generic',
});
```

The blueprint coordinate system is fixed: the nose points toward `-X`, `+Y` is
up, and `+Z` is the right span direction. Keep the root transform at identity;
hangar normalization, formation spacing, Task pose, and inspection rotation
belong to their scene runtimes.

## Evaluated external J-35 reference

The registry records the following optional replacement reference but does not
bundle it:

- **Shenyang J-35**, by SB-129
- Sketchfab model ID `37085f458d7943cd8314372af76e0f67`
- <https://sketchfab.com/3d-models/shenyang-j-35-37085f458d7943cd8314372af76e0f67>
- License shown by the source: Creative Commons Attribution 4.0
- Approximate source complexity: 8.2k triangles / 5.1k vertices

Sketchfab's download API requires an authenticated account token. The application
does not bypass that requirement or extract protected viewer payloads. If an
authorized GLB is later supplied, retain the author attribution and CC-BY 4.0
notice here and in the shipped product credits.

Do not add an aircraft asset unless its redistribution and commercial-use
terms are explicit. In particular, a non-commercial model is not a valid
replacement for this application.
