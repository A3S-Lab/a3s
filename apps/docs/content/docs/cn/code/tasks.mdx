# 任务

v2.1.0 的常规多智能体路径是 `task` 和 `parallel_task`，旧的公开 Team Runner API 不再使用。

`task` 适合单个有边界的子任务，`parallel_task` 适合互相独立的并行 lane。委派子运行应返回摘要、检查/修改过的文件、证据引用、风险和置信度，不应把完整 transcript 注入父 agent。

宿主已经知道任务边界时，可以直接调用 SDK helper：

```ts
await session.delegateTask({
  agent: 'explore',
  description: '检查认证模块',
  prompt: '返回检查过的文件、发现、风险和置信度。',
});

await session.parallelTask([
  { agent: 'explore', description: '检查配置', prompt: '检查 provider 解析。' },
  { agent: 'verification', description: '验证 SDK', prompt: '检查 SDK 声明。' },
]);
```

## Worker Agents

通过 `workerAgents` 或 `registerWorkerAgent()` 注册一次性 worker agents：

```ts
const session = agent.session('/repo', {
  workerAgents: [
    {
      name: 'frontend-cow',
      description: '小型前端修复',
      kind: 'implementer',
      model: 'openai/gpt-4o',
      maxSteps: 24,
      confirmationInheritance: 'auto_approve',
    },
  ],
});
```

### 确认继承

通过 `confirmationInheritance` 控制子运行如何处理 Ask 决策：

- `'auto_approve'`（默认）：子运行自动批准所有 Ask 决策
- `'deny_on_ask'`：子运行遇到 Ask 时立即失败
- `'inherit_parent'`：子运行继承父级的确认策略

```ts
const session = agent.session('/repo', {
  workerAgents: [
    {
      name: 'restricted-writer',
      description: '需要父级确认的文件写入',
      kind: 'implementer',
      confirmationInheritance: 'inherit_parent',  // 需要父级批准
    },
  ],
});
```

当子运行需要人工批准敏感操作时使用 `'inherit_parent'`，
或使用 `'deny_on_ask'` 确保子运行永远不会阻塞在确认提示上。

## Agent 目录

```ts
const session = agent.session('/repo', { agentDirs: ['./agents'] });
session.registerAgentDir('./more-agents');
```

常规团队协作请优先用 `task` / `parallel_task`。旧生命周期控制面 API 已移除；需要 UI 状态时，应用应消费 streaming events、run replay，以及 Node `cancelRun(runId)`。
