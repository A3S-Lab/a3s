# Agentic Search - 技术深度解析

本指南详细解释 `agentic_search` 工具背后的技术原理，并提供实用的实现示例。

<Callout type="info">
**灵感来源 Sirchmunk**：agentic_search 工具实现了受 [Sirchmunk 论文](https://github.com/modelscope/sirchmunk)启发的多阶段搜索管道，用智能关键词提取和相关性排序取代了传统的基于嵌入的向量搜索。

## 技术原理

### 架构概览

```
┌─────────────────────────────────────────────────────────────┐
│ 用户查询: "JWT token 验证是如何工作的"                       │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ 阶段 1: 关键词提取                                           │
│ - 移除停用词 ("是", "如何", "的")                          │
│ - 提取短语: ["JWT token", "token 验证"]                    │
│ - 提取单词: ["jwt", "token", "验证"]                       │
│ 结果: ["JWT token", "token 验证", "jwt", ...]              │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ 阶段 2: 并行搜索                                             │
│ ├─ 内容搜索: ripgrep 风格的正则匹配                        │
│ │  - 在文件内容中搜索关键词模式                            │
│ │  - 收集匹配行及其上下文                                  │
│ └─ 结构搜索: 文件名和路径匹配                              │
│    - 将关键词与文件名匹配                                  │
│    - 将关键词与目录路径匹配                                │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ 阶段 3: 相关性评分                                           │
│ - 基础分数: match_count / sqrt(file_lines)                │
│ - IDF 提升: unique_keywords_matched / total_keywords       │
│ - 文件类型提升: code(1.2x) > config(1.0x) > docs(0.9x)   │
│ - 最终分数: base_score * idf_boost * type_boost           │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ 阶段 4: 结果格式化                                           │
│ - 按相关性降序排序                                          │
│ - 格式化并添加上下文行                                      │
│ - 返回前 N 个结果                                           │
└─────────────────────────────────────────────────────────────┘
```

### 阶段 1: 关键词提取

关键词提取算法移除停用词，并在多个粒度上提取有意义的搜索词：

**算法：**
```rust
fn extract_keywords(query: &str) -> Vec<String> {
    let mut keywords = Vec::new();

    // 1. 提取 2-3 词组合
    let words: Vec<&str> = query.split_whitespace().collect();
    for window in words.windows(2) {
        keywords.push(window.join(" "));
    }

    // 2. 提取单个有意义的词（长度 >= 3）
    for word in &words {
        let clean = word.trim_matches(|c: char| !c.is_alphanumeric())
                        .to_lowercase();
        if clean.len() >= 3 && !STOP_WORDS.contains(&clean.as_str()) {
            keywords.push(clean);
        }
    }

    // 3. 去重并限制为前 8 个
    keywords.dedup();
    keywords.truncate(8);
    keywords
}
```

**示例：**
```
输入:  "JWT token 验证是如何工作的"
输出: ["JWT token", "token 验证", "jwt", "token", "验证"]
```

### 阶段 2: IDF 加权相关性评分

匹配更多唯一关键词的文件排名更高。这受 TF-IDF（词频-逆文档频率）启发，但针对代码搜索进行了简化。

**评分公式：**
```rust
// 基础分数：归一化匹配密度
let base_score = (match_count as f32) / (total_lines as f32).sqrt();

// IDF 提升：关键词覆盖率
let unique_keywords_matched = patterns
    .iter()
    .filter(|p| lines.iter().any(|l| p.is_match(l)))
    .count();
let idf_boost = (unique_keywords_matched as f32) / (patterns.len() as f32);

// 文件类型提升
let type_boost = match file_type {
    Code => 1.2,           // 优先代码文件
    Config => 1.0,         // 标准权重
    Documentation => 0.9,  // 稍低
    Other => 0.8,          // 最低优先级
};

// 最终相关性
let relevance = base_score * idf_boost * type_boost;
```

**为什么有效：**
- **基础分数**：较小、聚焦的文件比有分散匹配的大文件排名更高
- **IDF 提升**：匹配更多关键词的文件更相关
- **类型提升**：代码文件优先于文档

### 阶段 3: 蒙特卡洛证据采样（DEEP 模式）

DEEP 模式使用高斯重要性采样来探索高分匹配周围的代码区域。

**三幕策略：**

**第一幕：模糊锚定**
```rust
// 将高分区域识别为锚点
let anchors: Vec<usize> = file_matches
    .iter()
    .map(|m| m.line_number)
    .collect();
```

**第二幕：高斯重要性采样**
```rust
// 使用高斯权重在每个锚点周围采样
let sigma = (5.0 - match_density * 20.0).max(2.0);  // 自适应 sigma

for &anchor in &anchors {
    let window_radius = (sigma * 3.0) as usize;
    for line_idx in (anchor - window_radius)..(anchor + window_radius) {
        let distance = (line_idx as i32 - anchor as i32).abs();

        // 高斯权重: exp(-d² / (2σ²))
        let weight = (-((distance * distance) as f32) / (2.0 * sigma * sigma)).exp();

        sampled.push(SampledLine { line_idx, weight, ... });
    }
}
```

**第三幕：证据合成**
```rust
// 计算复合证据分数
let weight_sum: f32 = sampled.iter().map(|s| s.weight).sum();
let evidence_score = file_relevance * weight_sum / (sampled.len() as f32).sqrt();
```

**为什么使用高斯采样？**
- 聚焦于匹配周围的区域（而非随机采样）
- 衰减权重确保附近的行被优先考虑
- 自适应 sigma 根据匹配密度调整

## 性能特征

### 基准测试结果

在典型的中型项目（~50k 行代码）上：

| 操作 | 时间 | 说明 |
|------|------|------|
| 单关键词（FAST） | ~100ms | "authentication" |
| 多关键词（FAST） | ~150ms | "JWT token validation" |
| DEEP 模式 | ~15s | 蒙特卡洛采样 |
| 文件名搜索 | ~50ms | filename_only 模式 |
| 大型代码库（500k 行） | ~800ms | 使用 include 模式 |

### 优化技巧

1. **使用 include 模式**限制文件类型：
   ```
   include: "*.{rs,toml}"  // 只搜索 Rust 和 TOML 文件
   ```

2. **根据需求调整 max_results**：
   ```
   max_results: 3   // 快速概览
   max_results: 10  // 标准（默认）
   max_results: 20  // 全面
   ```

3. **使用 FILENAME_ONLY 进行发现**：
   ```
   mode: "filename_only"  // 比内容搜索快 10 倍
   ```

4. **为复杂查询保留 DEEP 模式**：
   ```
   mode: "deep"  // 仅当 FAST 模式不足时使用
   ```

## 故障排除

### 搜索太慢

**问题：**搜索耗时 > 5 秒

**解决方案：**
1. 使用 `include` 模式限制文件类型
2. 减少 `max_results`
3. 使用 `filename_only` 模式进行快速发现

### 找不到预期结果

**问题：**相关文件在结果中缺失

**解决方案：**
1. 尝试更具体的关键词
2. 检查文件是否被 `.gitignore` 排除
3. 验证文件大小是否 < 1MB（跳过大文件）
4. 尝试使用 `grep` 进行精确匹配

### 结果太多

**问题：**结果过于庞大

**解决方案：**
1. 使查询更具体
2. 使用 `include` 模式过滤文件类型
3. 减少 `max_results`
4. 减少 `context_lines`
