# 多智能体编排

A3S Code 提供四种多智能体协作模式，从轻量委派到完全可控的团队编排：

| 模式 | API | 适用场景 |
|------|-----|---------|
| LLM 内委派 | `task` / `parallel_task` 工具 | LLM 自主决定何时将工作分配给子代理 |
| 独立子代理 | `spawn(AgentSlot)` | 从代码层面启动单个子代理，支持实时监控和控制 |
| 团队工作流 | `run_team(goal, ws, slots)` | 自动组建 Lead → Worker → Reviewer 团队并协调执行 |
| 低级任务板 | `AgentTeam` + `TeamRunner` 直接操作 | 需要完全控制编排逻辑的高级场景 |

## 架构

```text
┌─────────────────────────────────────────────────────────┐
│                    AgentOrchestrator                    │
│                                                         │
│  ┌──────────────────┐      ┌─────────────────────────┐  │
│  │  spawn(AgentSlot) │      │ run_team(goal, ws, slots)│  │
│  │  role = None      │      │  role = Lead/Worker/     │  │
│  │                   │      │         Reviewer         │  │
│  └────────┬─────────┘      └──────────────┬──────────┘  │
│           │                               │              │
│  SubAgentWrapper                    AgentTeam +          │
│  ├─ AgentRegistry                   TeamRunner           │
│  ├─ ToolExecutor                    ├─ Lead 分解目标     │
│  └─ AgentLoop (真实 LLM)            ├─ Worker 执行任务   │
│                                     └─ Reviewer 审批     │
└─────────────────────────────────────────────────────────┘
              ↑
        AgentRegistry
   (内置: general / explore / plan)
   (自定义: ~/.a3s/agents/*.yaml)
```

**三个核心概念：**

| 概念 | 职责 |
|------|------|
| `AgentDefinition` / `AgentRegistry` | 静态 agent 定义（YAML/Markdown），描述 agent 的提示词、权限和工具访问 |
| `AgentSlot` | 运行时声明类型，指定要启动的 agent 类型和角色 |
| `AgentOrchestrator` | 执行引擎，通过 `spawn` 或 `run_team` 驱动 AgentLoop |

### 自定义子代理

在 agent 目录中创建 YAML 或 Markdown 文件，系统启动时自动加载：

```hcl

agent_dirs = ["~/.a3s/agents", "/shared/team-agents"]
```

**YAML 格式：**

```yaml
# ~/.a3s/agents/security-reviewer.yaml
name: security-reviewer
description: Reviews code for security vulnerabilities and best practices
mode: subagent
hidden: false
max_steps: 25
can_spawn_subagents: false

permissions:
  allow:
    - read
    - grep
    - glob
    - ls
  deny:
    - write
    - edit
    - bash
    - patch

prompt: |
  You are a security code reviewer. Analyze the codebase for:
  - SQL injection, XSS, CSRF vulnerabilities
  - Hardcoded secrets and credentials
  - Insecure cryptographic practices
  Report findings with severity (Critical/High/Medium/Low),
  affected file and line, and recommended fix.
  Never modify any files — report only.
```

**Markdown 格式**（YAML frontmatter + 正文成为系统提示）：

```markdown
---
name: db-migration
description: Database migration specialist
mode: subagent
max_steps: 40
permissions:
  allow: [read, write, edit, bash, grep, glob]
  deny:  [web_fetch, web_search]
---
You are a database migration specialist...
```

通过 `model` 字段为特定 agent 指定不同模型：

```yaml
name: quick-search
model: anthropic/claude-haiku-4-20250514

  allow: [read, grep, glob, ls]
```

## AgentSlot — 统一声明类型

`AgentSlot` 是描述"如何启动一个 agent"的统一声明类型，同时用于独立子代理和团队成员：

```rust
pub struct AgentSlot {
    pub agent_type: String,             // "general" | "explore" | "plan" | 自定义名称
    pub role: Option<TeamRole>,         // None = 独立子代理; Some(Lead/Worker/Reviewer) = 团队成员
    pub description: String,
    pub prompt: String,
    pub permissive: bool,               // true = 跳过人工确认
    pub max_steps: Option<usize>,
    pub timeout_ms: Option<u64>,
    pub workspace: String,
    pub agent_dirs: Vec<String>,        // 额外的自定义 agent 目录
    // ... 其他字段
}
```

`role` 字段决定 slot 的执行路径：
- `role = None` → `spawn()` 作为独立子代理运行
- `role = Some(Lead)` / `Some(Worker)` / `Some(Reviewer)` → `run_team()` 组建团队

## 路径一：独立子代理 (spawn)

`spawn(AgentSlot)` 在隔离的子会话中运行单个 agent，支持实时监控和动态控制。

### 快速开始

<Tabs items={['Rust', 'TypeScript', 'Python']}>
<Tab value="Rust">

```rust
use a3s_code_core::{Agent, orchestrator::{AgentOrchestrator, AgentSlot}};
use std::sync::Arc;

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let agent = Arc::new(Agent::new("agent.hcl").await?);
    let orch = AgentOrchestrator::from_agent(agent);

    let slot = AgentSlot::new("general", "查找所有 Rust 源文件")
        .with_description("Find Rust files")
        .with_permissive(true)
        .with_max_steps(10);

    let handle = orch.spawn(slot).await?;
    let result = handle.wait().await?;
    println!("{result}");
    Ok(())
}
```

## 路径二：团队工作流 (run_team)

`run_team(goal, workspace, slots)` 自动组建 Lead → Worker → Reviewer 团队，将目标分解成任务并协调执行。

<Callout type="info">
  **前提**：需要使用 `Orchestrator.create(agent)` 挂载真实 Agent。
</Callout>

### 快速开始

<Tabs items={['Rust', 'TypeScript', 'Python']}>
<Tab value="Rust">

```rust
use a3s_code_core::{Agent, orchestrator::{AgentOrchestrator, AgentSlot, TeamRole}};
use std::sync::Arc;

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let agent = Arc::new(Agent::new("agent.hcl").await?);
    let orch = AgentOrchestrator::from_agent(agent);

    let slots = vec![
        AgentSlot::new("general", "").with_role(TeamRole::Lead).with_max_steps(5),
        AgentSlot::new("general", "").with_role(TeamRole::Worker).with_max_steps(5),
        AgentSlot::new("general", "").with_role(TeamRole::Reviewer).with_max_steps(3),
    ];

    let result = orch.run_team(
        "列举 3 种常见的 Rust 数据结构并简要说明",
        ".",
        slots,
    ).await?;

    println!("完成任务: {}", result.done_tasks.len());
    println!("拒绝任务: {}", result.rejected_tasks.len());
    println!("轮数: {}", result.rounds);
    for task in &result.done_tasks {
        println!("  [{}] {}", task.id, task.result.as_deref().unwrap_or(""));
    }
    Ok(())
}
```

### 工作流程

```
1. Lead 分解
   ├─ 接收目标，输出 JSON: {"tasks": ["任务1", "任务2", ...]}
   └─ 为每个任务调用 board.post()

2. Worker 执行（并发）
   ├─ 循环 claim() → 认领任务 → session.send(任务) → board.complete()
   └─ 无任务时等待 poll_interval_ms

3. Reviewer 审批
   ├─ 对每个 InReview 任务，检查结果是否符合要求
   ├─ 输出 "APPROVED: <原因>" → board.approve() → 任务进入 Done
   └─ 输出 "REJECTED: <反馈>" → board.reject() → 任务重新入队

4. 终止条件
   └─ 全部任务完成，或达到 max_rounds 上限
```

### TeamRole 枚举

| 角色 | Rust | TypeScript/Python | 职责 |
|------|------|-------------------|------|
| Lead | `TeamRole::Lead` | `"lead"` | 分解目标，发布任务 |
| Worker | `TeamRole::Worker` | `"worker"` | 认领并执行任务 |
| Reviewer | `TeamRole::Reviewer` | `"reviewer"` | 审批任务结果 |

## 路径三：低级任务板

对于需要完全控制编排逻辑的场景，直接操作 `AgentTeam` + `TeamTaskBoard` + `TeamRunner`。

### TeamTaskBoard

任务板通过 `Arc<TeamTaskBoard>` 在所有成员之间共享，所有操作均线程安全：

```rust
let board = team.task_board();     // &TeamTaskBoard
let board = team.task_board_arc(); // Arc<TeamTaskBoard>

// 发布任务 — 返回 Some(task_id) 或 None（任务板已满）
let id = board.post("重构 JWT 验证", "lead", None).unwrap();
// 预指派给特定 Worker；任务直接进入 InProgress 状态
let id = board.post("添加 CSRF 防护", "lead", Some("worker-1")).unwrap();

// 认领下一个 Open 或 Rejected 状态的任务
let task: Option<TeamTask> = board.claim("worker-1");

// 完成任务（移至 InReview 状态）
board.complete(&task.id, "已添加 RS256 密钥轮换");

// Reviewer 操作
board.approve(&task.id);   // InReview → Done
board.reject(&task.id);    // InReview → Rejected（可被 Worker 重新认领）

// 查询
let tasks: Vec<TeamTask>    = board.by_status(TaskStatus::Done);
let tasks: Vec<TeamTask>    = board.by_assignee("worker-1");
let task:  Option<TeamTask> = board.get(&id);

// 统计 — (open, in_progress, in_review, done, rejected)
let (open, prog, rev, done, rej) = board.stats();
let total = board.len();
```

### 任务状态机

```
post()      claim()       complete()
  Open ──► InProgress ──► InReview
                               │
                   ┌───────────┴──────────┐
              approve()              reject()
                   │                      │
                 Done           Rejected ──► (重新 claim())
```

### TeamRunner

`TeamRunner` 将真实的 `AgentSession`（或任意 `AgentExecutor`）绑定到团队成员，并运行完整的自动化工作流：

```rust
use a3s_code_core::agent_teams::TeamRunner;
use std::sync::Arc;

let mut runner = TeamRunner::new(team); // team 在此处被消耗

runner.bind_session("lead",     Arc::new(agent.session(".", None)?))?;
runner.bind_session("worker-1", Arc::new(agent.session(".", None)?))?;
runner.bind_session("reviewer", Arc::new(agent.session(".", None)?))?;

let result = runner.run_until_done("审查 auth 模块").await?;
```

`run_until_done` 内部工作流：

1. **Lead 分解** — Lead Session 收到以下提示，Runner 解析 JSON 并调用 `board.post()`：
   ```
   You are the lead agent in a team. Your goal is: {goal}
   Decompose this goal into concrete, self-contained tasks for your team workers.
   Respond with ONLY valid JSON: {"tasks": ["task 1", "task 2", ...]}
   ```

2. **Worker 执行** — 每个 Worker 启动独立的 Tokio 任务，循环 `claim() → session.send() → board.complete()`

3. **Reviewer 循环** — 对每个 InReview 任务，Reviewer Session 收到任务描述和结果，回应 `APPROVED: <原因>` 或 `REJECTED: <反馈>`

4. **终止条件** — 所有任务完成，或达到 `max_rounds` 上限

### AgentExecutor Trait


## LLM 内委派

在 session 内部，LLM 通过 `task` 和 `parallel_task` 工具自主将工作委派给子代理。`delegate-task` 内置 Skill（Instruction 类型）会自动注入到系统提示中，教会 LLM 何时委派、哪些 agent 可用以及如何调用工具——无需额外配置。

### 委派工作原理

```
父会话
  │
  ├─ task("security-reviewer", "Review auth module")
  │    │
  │    ├─ 1. 在 AgentRegistry 中查找 AgentDefinition
  │    ├─ 2. 使用 agent 的权限创建子会话
  │    ├─ 3. 应用 agent 的系统提示
  │    ├─ 4. 在隔离的 agentic 循环中执行任务
  │    ├─ 5. 将结果返回给父会话
  │    └─ 6. 保留子会话 ID 以供引用
  │
  └─ 父会话继续使用子代理的结果
```

关键隔离保证：
- 子会话有独立的上下文窗口（无父上下文泄漏）
- 权限限制为 agent 定义的策略
- `max_steps` 防止失控执行
- `can_spawn_subagents: false` 防止递归委派

### 子代理事件

流式传输时，子代理生命周期事件会在父会话的事件通道上发出：

```rust
let (mut rx, _handle) = session.stream("全面分析代码库").await?;
while let Some(event) = rx.recv().await {
    match event {
        AgentEvent::SubagentStart { task_id, agent, description, .. } => {
            println!("▶ [{agent}] {description} (task: {task_id})");
        }
        AgentEvent::SubagentEnd { agent, success, output, .. } => {
            let status = if success { "✓" } else { "✗" };
            println!("{status} [{agent}] {}", output.chars().take(100).collect::<String>());
        }
        AgentEvent::End { .. } => break,
        _ => {}
    }
}
```

## 多会话并行

对于超出子代理委派范围的工作负载——例如在多个仓库或不同工作区运行相同分析——可以创建多个独立会话，使用语言原生的并发原语驱动它们。

<Tabs items={['Rust', 'TypeScript', 'Python']}>
<Tab value="Rust">

```rust
use a3s_code_core::Agent;
use tokio::task::JoinSet;

let agent = Agent::new("agent.hcl").await?;

let workspaces = vec![
    "/repos/service-auth",
    "/repos/service-billing",
    "/repos/service-gateway",
];

// 为每个工作区生成一个 session，全部并发运行
let mut join_set = JoinSet::new();
for workspace in workspaces {
    let agent = agent.clone();
    join_set.spawn(async move {
        let session = agent.session(workspace, None)?;
        let result = session.send("查找所有 TODO 注释并汇总").await?;
        Ok::<_, a3s_code_core::CodeError>((workspace.to_string(), result.text))
    });
}

// 按完成顺序收集结果
while let Some(result) = join_set.join_next().await {
    match result? {
        Ok((workspace, text)) => println!("=== {workspace} ===\n{text}\n"),
        Err(e) => eprintln!("Error: {e}"),
    }
}
```

### 扇出/扇入模式

将多会话并行与子代理委派结合，构建扇出/扇入工作流：

```rust
use a3s_code_core::Agent;
use tokio::task::JoinSet;

let agent = Agent::new("agent.hcl").await?;

// 阶段一：跨工作区并行探索
let mut join_set = JoinSet::new();
for workspace in &["/repos/frontend", "/repos/backend", "/repos/infra"] {
    let agent = agent.clone();
    let ws = workspace.to_string();
    join_set.spawn(async move {
        let session = agent.session(&ws, None)?;
        // 每个 session 内部再用 parallel_task 进一步扇出
        let result = session.tool("parallel_task", serde_json::json!({
            "tasks": [
                { "agent": "explore",           "description": "查找依赖", "prompt": "列出所有外部依赖。" },
                { "agent": "security-reviewer", "description": "安全扫描", "prompt": "扫描已知漏洞模式。" }
            ]
        })).await?;
        Ok::<_, a3s_code_core::CodeError>((ws, result.output))
    });
}

// 阶段二：收集结果
let mut findings = Vec::new();
while let Some(result) = join_set.join_next().await {
    if let Ok(Ok((ws, output))) = result {
        findings.push(format!("## {ws}\n{output}"));
    }
}

// 阶段三：汇总到单个 session
let summary_session = agent.session("/repos", None)?;
let summary = summary_session.send(&format!(
    "将这些发现汇总成一份安全报告：\n\n{}",
    findings.join("\n\n")
)).await?;

println!("{}", summary.text);
```

## Orchestrator 创建模式

| 模式 | API | 子代理运行方式 |
|------|-----|---------------|
| 占位符（测试） | `Orchestrator.create()` / `AgentOrchestrator::new_memory()` | 内部 stub，不调用 LLM |
| 真实执行 | `Orchestrator.create(agent)` / `AgentOrchestrator::from_agent(agent)` | 完整 AgentLoop，含真实 LLM + 工具 |

## 模式选择指南

| 场景 | 推荐模式 |
|------|---------|
| LLM 自主决定是否需要委派 | LLM 内委派（task / parallel_task） |
| 从代码层运行单个专用任务 | `spawn(AgentSlot)` |
| 需要实时监控和暂停/恢复控制 | `spawn(AgentSlot)` |
| 目标需要分解成多个子任务并行执行 | `run_team(goal, ws, slots)` |
| 需要 Lead/Worker/Reviewer 协作和质量审批 | `run_team(goal, ws, slots)` |
| 需要完全控制任务板和编排逻辑 | `AgentTeam` + `TeamRunner` 直接操作 |
| 多仓库或跨工作区并行分析 | 多会话并行（JoinSet / Promise.all） |
