# 团队

团队现在是一种驾驭层模式：命名 agent 定义加上 `task` 与 `parallel_task`。

v2.1.0 不再暴露旧公开 team API。推荐做法是把角色写成 agent 定义文件，用父 session 的 `agentDirs` 加载，然后让父 agent 通过 `task` 或 `parallel_task` 委派。

```ts
const session = agent.session('/repo', {
  agentDirs: ['./agents'],
  builtinSkills: true,
  planningMode: 'enabled',
});

await session.send(`
组织发布检查团队：
- explorer: 找出高风险变更区域
- tester: 找出缺失验证
- security: 检查副作用路径
使用 parallel_task，并合并成一份阻塞项优先的报告。
`);
```

如果宿主已经知道需要哪些 lane，也可以直接调用 `session.delegateTask(...)`
或 `session.parallelTask(...)`。

## Worker Agents

通过 `workerAgents` 或 `registerWorkerAgent()` 注册一次性 worker agents：

```ts
const session = agent.session('/repo', {
  workerAgents: [
    {
      name: 'frontend-cow',
      description: '小型前端修复',
      kind: 'implementer',
      model: 'openai/gpt-4o',
      maxSteps: 24,
      confirmationInheritance: 'auto_approve',
    },
  ],
});
```

### 确认继承

通过 `confirmationInheritance` 控制子运行如何处理 Ask 决策：

- `'auto_approve'`（默认）：子运行自动批准所有 Ask 决策
- `'deny_on_ask'`：子运行遇到 Ask 时立即失败
- `'inherit_parent'`：子运行继承父级的确认策略

```ts
const session = agent.session('/repo', {
  workerAgents: [
    {
      name: 'restricted-writer',
      description: '需要父级确认的文件写入',
      kind: 'implementer',
      confirmationInheritance: 'inherit_parent',  // 需要父级批准
    },
  ],
});
```

当子运行需要人工批准敏感操作时使用 `'inherit_parent'`，
或使用 `'deny_on_ask'` 确保子运行永远不会阻塞在确认提示上。

## 运行状态

旧生命周期控制面 API 已移除。应用应通过 streaming events 和 run replay
快照展示状态，Node 侧需要取消时使用 `cancelRun(runId)`。
