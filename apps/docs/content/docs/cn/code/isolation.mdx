# 隔离策略

A3S Code 提供四层隔离机制，防止 agent 的操作影响到不应被修改的文件或系统资源。

```
隔离层次
  ├─ 工作区边界     ← 文件操作限制在指定目录内（始终生效）
  ├─ 文件版本历史   ← 写入前自动快照，支持 diff 与恢复
  ├─ Git 工作树     ← agent 在独立分支上操作，出错可整个丢弃
  └─ MicroVM 沙箱   ← bash 命令在隔离虚拟机内执行（宿主应用提供 BashSandbox 实现）
```

## 工作区边界

所有文件操作（`read`、`write`、`edit`、`glob`、`grep`）均被限制在 session 创建时指定的工作区目录内。尝试访问工作区外的路径会被直接拦截，不会产生错误重试，也不会向 LLM 反馈路径内容。

这一边界**始终生效**，无法通过配置关闭。

## 文件版本历史

每当 agent 执行 `write`、`edit` 或 `patch` 工具时，A3S Code 自动在修改前捕获文件快照。快照保存在会话内存中，支持版本列表、diff 生成和恢复。

### 自动快照时机

| 工具 | 触发快照 |
|------|---------|
| `write` | 覆写文件前 |
| `edit` | 字符串替换前 |
| `patch` | 应用补丁前 |

### 查看版本历史

<Tabs groupId="lang" items={['TypeScript', 'Python']}>
<Tab value="TypeScript">
```typescript
// 列出文件的所有版本
const versions = await session.listFileVersions('src/auth.rs');
for (const v of versions) {
  console.log(`v${v.version} — ${v.toolName} — ${v.timestamp}`);
}

// 生成 diff
const diff = await session.diffFileVersions('src/auth.rs', 0, 1);
console.log(diff);

// 恢复到版本 0
await session.restoreFileVersion('src/auth.rs', 0);
```
</Tab>

## Git 工作树

`git_worktree` 工具让 agent 在独立的 Git 工作树（分支）上工作。agent 的所有改动都在隔离分支上，主分支完全不受影响。如果实验失败，直接删除工作树即可，一行命令清除所有修改。

### 工作流

<Steps>
<Step>**创建工作树** — 在新分支上建立隔离工作区</Step>
<Step>**在隔离分支上工作** — agent 的所有文件修改都提交到该分支</Step>
<Step>**验证结果** — 检查修改，决定是合并还是丢弃</Step>
<Step>**合并或丢弃** — 满意则 merge，不满意则 `remove` 整个工作树</Step>
</Steps>

### 使用示例

<Tabs groupId="lang" items={['TypeScript', 'Python']}>
<Tab value="TypeScript">
```typescript
// 创建隔离工作树
await session.callTool('git_worktree', {
  command: 'create',
  branch: 'agent/refactor-auth',
  path: '/tmp/agent-worktrees/refactor-auth',
  newBranch: true,
  base: 'main',
});

// 在隔离工作树上创建新 session
const isolatedSession = agent.session('/tmp/agent-worktrees/refactor-auth');

// agent 的所有修改都在隔离分支上
const result = await isolatedSession.send('重构 auth 模块使用 JWT');

// 查看所有工作树
const list = await session.callTool('git_worktree', { command: 'list' });

// 不满意则删除
await session.callTool('git_worktree', {
  command: 'remove',
  path: '/tmp/agent-worktrees/refactor-auth',
});
```
</Tab>

## MicroVM 沙箱

当 `bash` 工具的命令需要在 A3S Box MicroVM 内执行（而非宿主机进程）时，宿主应用通过 `SessionOptions::with_sandbox_handle()` 提供一个实现了 `BashSandbox` trait 的具体实现。工作区目录以读写方式挂载到 MicroVM 内的 `/workspace`。

<Callout type="info">
无需任何 Cargo feature 标志。`a3s-code-core` 将 `BashSandbox` trait 作为开放扩展点对外暴露。具体的沙箱实现由宿主应用（如 SafeClaw）提供。
</Callout>

### Rust — 接入自定义沙箱

```rust
use a3s_code::{sandbox::BashSandbox, agent_api::SessionOptions};
use std::sync::Arc;

// 任何实现了 BashSandbox 的类型都可以在此传入。
// SafeClaw 内置了基于 a3s-box 的实现；你也可以提供自己的实现。
let opts = SessionOptions::default()
    .with_sandbox_handle(Arc::new(my_sandbox_impl));

let session = agent.session("/project", opts).await?;
```

### BashSandbox trait

```rust
#[async_trait]
pub trait BashSandbox: Send + Sync {
    /// 在沙箱内执行 shell 命令。
    async fn run(&self, command: &str, config: &SandboxConfig) -> Result<SandboxOutput>;
}
```

## 选择隔离方案

| 场景 | 推荐方案 |
|------|---------|
| 防止读取项目外文件 | 工作区边界（始终生效） |
| agent 写文件后想撤销 | 文件版本历史 |
| agent 在实验性分支工作 | Git 工作树 |
| 需要隔离 bash 命令的系统副作用 | MicroVM 沙箱 |
| 生产环境高安全需求 | Git 工作树 + MicroVM 沙箱 组合 |
