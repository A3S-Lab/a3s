# A3S Observer

`a3s-observer` 把内核级事件翻译成语义化的 AI Agent 遥测 —— **谁**(进程 / k8s pod)跑了
**什么**(工具、文件、LLM 调用)、连到**哪里**(端点) —— 用 **eBPF** 捕获,**零改 agent
代码**、跨语言。

```
捕获 (eBPF 探针)            富化 + 关联 (用户态)                 导出
─────────────────         ───────────────────────────         ──────────
exec    (子进程)           identity: k8s cgroup→pod,            NDJSON
file    (写文件)                      /proc + 内核态 comm        (或日志)
connect (peer IP)         (pid,fd)→peer 关联                   → OTel
tls-sni (LLM 厂商)         LLM 请求/响应字节·延迟·TTFT             Collector
dns     (主机名)
```

一条事件就回答 谁 / 什么 / 哪里 —— 这里是一次带指标的 LLM 调用:

```json
{"identity":{"agent":"curl"},"provider":"Anthropic",
 "event":{"LlmCall":{"sni":"api.anthropic.com","peer":"160.79.104.10",
   "req_bytes":284,"resp_bytes":3832,
   "latency":{"secs":1,"nanos":20000000},"ttft":{"secs":0,"nanos":318000000}}}}
```

## 为什么用 eBPF(而不是 SDK)

- **零侵入、语言无关** —— 不碰 agent 代码或 SDK,就能观测任意运行时(Python / Node / Go / Rust)。
- **看到 app 不会告诉你的** —— 真实的子进程 exec、文件写入、网络外联,包括 agent 派生的工具子进程。
- **低开销、与安全相关** —— 跑在内核里,agent 难以规避。

<Callout type="info">
v1 只用语言无关的内核钩子(无按语言的 uprobe),因此对任意运行时都生效。代价:拿不到 LLM
prompt / model 名 / 精确 token —— 那需要一个可选的 TLS-payload 扩展(路线图)。默认捕获厂商、
端点、请求/响应字节、延迟、TTFT、工具、文件与身份。
</Callout>

## 它捕获什么

| 探针 | 钩子 | 给你 |
|---|---|---|
| `exec` | `sys_enter_execve` | 工具 / 子进程(argv)—— 包括绕过 app 插桩的那些 |
| `file` | `sys_enter_openat`(写打开) | agent **写**的文件 —— 可选,`A3S_OBSERVER_FILES=1`(繁忙节点上量很大) |
| `connect` | `sys_enter_connect` | peer **IP:端口** |
| `tls-sni` | 发送时的 TLS ClientHello | LLM **厂商**(来自 SNI),语言无关 |
| `dns` | `sendto` / `sendmsg` / `sendmmsg` :53 | 解析的**主机名** |
| LLM 指标 | 每个 socket 的 `read` / `recv` + `close` | 请求/响应**字节**、**延迟**、**TTFT** |

用户态给每条事件富化**身份**(k8s pod/容器 via cgroup,或 `/proc` comm+ppid,并以内核态
`comm` 兜底,使短命进程也能被归属)以及一个 `(pid,fd)→peer` 关联,把 LLM 厂商与其端点、指标
融合到一起。

## 架构(最小核心 + 扩展)

探针核心外围是可替换的 trait:

- `IdentityResolver` —— k8s cgroup→pod · `/proc`(裸机) · 内核态 `comm` 兜底
- `ServiceClassifier` —— SNI / IP → `Provider`(14 个 LLM 厂商)
- `Exporter` —— NDJSON · 日志 → OpenTelemetry Collector(filelog → OTLP)

Crates:`a3s-observer`(契约 + 模型)、`a3s-observer-common`(`no_std` 共享类型)、
`a3s-observer-ebpf`(探针)、`a3s-observer-collector`(加载器 + 关联 + 导出)。

## 构建与运行

eBPF crate 需要 nightly + `rust-src` + `bpf-linker`(它借用 rustc 自带的 LLVM —— **无需系统
LLVM**):

```bash
rustup toolchain install nightly --component rust-src
cargo install bpf-linker
cargo build --release -p a3s-observer-collector

# 仅 Linux;需 root / CAP_BPF
sudo ./target/release/a3s-observer-collector                          # 人类可读日志
A3S_OBSERVER_JSON=1 sudo -E ./target/release/a3s-observer-collector   # NDJSON → vector/Loki/jq
```

## 导出到 OpenTelemetry

a3s-observer 保持精简 —— 它只产 NDJSON;运输(批处理 / 重试 / 路由)是 OpenTelemetry
Collector 的活。生产管道:

```
捕获 (a3s-observer)  →  NDJSON  →  OTel Collector (filelog → OTLP)  →  你的后端
```

现成配置在 `deploy/otel-collector.yaml`。在 Kubernetes 里,把 a3s-observer 作为 DaemonSet
输出到 stdout,让 Collector DaemonSet 去 tail 节点日志。进程内 OTLP 推送**刻意不做** —— 那是
Collector 的活(也让特权探针二进制保持最小、与后端可用性解耦)。

## 状态与路线图

**已实现**(在 Linux 6.8 验证):`exec` / `file`(写) / `connect` / `tls-sni` / `dns` 探针 +
每 socket 的 **LLM 指标**(请求/响应字节、延迟、TTFT);身份(k8s、`/proc`、内核态 `comm`);
`(pid,fd)→peer` 关联;NDJSON / 日志导出 → OpenTelemetry Collector。

**路线图:** 可选的 **SSL-payload** 扩展用于 prompt/响应内容(按 TLS 库实现,因为它需要非语言
无关的 uprobe);a3s-box MicroVM 的 in-guest 探针。
