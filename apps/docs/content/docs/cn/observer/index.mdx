# A3S Observer

`a3s-observer` 把内核级活动翻译成语义化 AI Agent 遥测：哪个进程或 pod 跑了哪个工具、
碰了哪个文件、调用了哪个 LLM 厂商、连到了哪个端点、触发了哪个安全敏感 syscall。
它可以跨语言观测未改造的 agent，也可以通过外部策略文件接入内核 guard，选择性拒绝
egress、file access 或 exec。

```
agent + tool subprocesses
        |
        | exec, exit, connect, DNS, TLS ClientHello, file, unlink, security syscalls
        v
kernel probes / fanotify
        |
        v
collector: identity + pid/fd correlation + provider classification
        |
        v
NDJSON -> OpenTelemetry Collector / Sentry / your backend
```

## 为什么用 eBPF(而不是 SDK)

- **零侵入、语言无关** —— 不碰 agent 代码或 SDK,就能观测任意运行时(Python / Node / Go / Rust)。
- **看到 app 不会告诉你的** —— 真实的子进程 exec、文件写入、网络外联,包括 agent 派生的工具子进程。
- **低开销、与安全相关** —— 跑在内核里,agent 难以规避。

<Callout type="info">
默认 observe path 是被动的。干预能力是 opt-in 且隔离的：`a3s-observer-enforce`
通过 cgroup BPF 处理 egress，`a3s-observer-fileguard` 通过 fanotify 处理 file 和 exec denial。
</Callout>

## 当前能力

| 领域 | 已实现能力 |
| --- | --- |
| 被动遥测 | `ToolExec`、`ProcessExit`、`Egress`、`Dns`、`LlmCall`、`SecurityAction`、collector heartbeat，以及基于 SNI 的 provider classification。 |
| 可选内容捕获 | `A3S_OBSERVER_SSL=1` 启用 OpenSSL `SSL_write` / `SSL_read` uprobes，产出 `SslContent`；能解析时会从中生成 `LlmApi`。 |
| 可选文件信号 | `A3S_OBSERVER_FILES=1` 启用 file write 与 delete 信号。 |
| 身份 | Kubernetes cgroup 到 pod 身份、`/proc` 进程 metadata、短命进程的内核 `comm` 兜底。 |
| 关联 | 用户态 `(pid, fd) -> peer` 关联，把 socket、provider、endpoint 和 LLM metrics 连起来。 |
| 导出 | NDJSON 或人类日志；生产运输交给 OpenTelemetry Collector 或其它日志管道。 |
| 干预 | `a3s-observer-enforce` 处理 egress deny；`a3s-observer-fileguard` 处理 file/exec deny；两者都由外部热加载策略文件驱动。 |

## 架构(最小核心 + 扩展)

探针核心外围是可替换的 trait:

- `IdentityResolver` —— k8s cgroup→pod · `/proc`(裸机) · 内核态 `comm` 兜底
- `ServiceClassifier` —— SNI / IP → `Provider`(15 个 LLM 厂商)
- `Exporter` —— NDJSON · 日志 → OpenTelemetry Collector(filelog → OTLP)
- `Policy` —— 可选的 host-buildable decision seam，用于 egress、file 和 exec verdict

Crates:`a3s-observer`(契约 + 模型)、`a3s-observer-common`(`no_std` 共享类型)、
`a3s-observer-ebpf`(探针)、`a3s-observer-collector`(加载器、关联、导出、`enforce`
和 `fileguard`)。

## 构建和运行

eBPF crate 需要 nightly + `rust-src` + `bpf-linker`(它借用 rustc 自带的 LLVM —— **无需系统
LLVM**):

```bash
rustup toolchain install nightly --component rust-src
cargo install bpf-linker
cargo build --release -p a3s-observer-collector

# 仅 Linux;需 root / CAP_BPF
sudo ./target/release/a3s-observer-collector                          # 人类可读日志
A3S_OBSERVER_JSON=1 sudo -E ./target/release/a3s-observer-collector   # NDJSON → vector/Loki/jq
```

仅 Linux；需要 root 或对应能力（`CAP_BPF`、`CAP_PERFMON`，以及 fanotify/guard 路径需要的权限）。

## 边界

- Always-on core 是被动的。SSL content 和 file events 是 opt-in，因为它们更高流量、
  更敏感。
- OpenSSL plaintext capture 只覆盖使用 OpenSSL 符号的进程；Go `crypto/tls` 路径不在该
  uprobe 路径内。
- A3S Box guest 有自己的 kernel。host-side Observer 可以看到 guest 通过 host network
  path 的 egress，但 in-guest exec/file telemetry 需要 guest 内 collector。
