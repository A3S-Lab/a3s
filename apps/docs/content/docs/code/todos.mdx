# Todo Tracking

Each session maintains a todo list for tracking multi-step tasks. The agent creates, updates, and completes todos as it works through complex requests.

## Todo Structure

Todos use the unified Task model shared with the [planning system](/docs/code/planning).

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique identifier |
| `title` | string | Short task description |
| `description` | string | Detailed task description |
| `status` | TaskStatus | Current status |
| `priority` | enum | `high`, `medium`, `low` |
| `dependencies` | string[] | Task IDs that must complete first |
| `created_at` | timestamp | When the todo was created |
| `completed_at` | timestamp | When the todo was completed |

### TaskStatus

| Status | Description |
|--------|-------------|
| `Pending` | Not yet started |
| `InProgress` | Currently being executed |
| `Completed` | Successfully finished |
| `Failed` | Execution failed |
| `Skipped` | Skipped (dependency failed or no longer needed) |
| `Cancelled` | Manually cancelled |

## How It Works

When the agent receives a complex task, it may decompose it into todos:

## Context Integration

The todo list is included in the agent's context, so it always knows what work remains. This prevents the agent from losing track of subtasks in long-running sessions.

Todos persist with the session ‚Äî if a session is restored from storage, the todo list is restored too.
