# Multi-Agent Orchestration

A3S Code provides four multi-agent collaboration modes, from lightweight delegation to fully controllable team orchestration:

| Mode | API | Use when |
|------|-----|----------|
| LLM-driven delegation | `task` / `parallel_task` tools | LLM autonomously decides when to delegate work to subagents |
| Standalone subagent | `spawn(AgentSlot)` | Launch a single subagent from code with real-time monitoring and control |
| Team workflow | `run_team(goal, ws, slots)` | Automatically assemble Lead → Worker → Reviewer team and coordinate execution |
| Direct task board | `AgentTeam` + `TeamRunner` | Advanced scenarios requiring full control over orchestration logic |

## Architecture

```text
┌─────────────────────────────────────────────────────────┐
│                    AgentOrchestrator                    │
│                                                         │
│  ┌──────────────────┐      ┌─────────────────────────┐  │
│  │  spawn(AgentSlot) │      │ run_team(goal, ws, slots)│  │
│  │  role = None      │      │  role = Lead/Worker/     │  │
│  │                   │      │         Reviewer         │  │
│  └────────┬─────────┘      └──────────────┬──────────┘  │
│           │                               │              │
│  SubAgentWrapper                    AgentTeam +          │
│  ├─ AgentRegistry                   TeamRunner           │
│  ├─ ToolExecutor                    ├─ Lead decomposes   │
│  └─ AgentLoop (real LLM)            ├─ Workers execute   │
│                                     └─ Reviewer approves │
└─────────────────────────────────────────────────────────┘
              ↑
        AgentRegistry
   (built-in: general / explore / plan)
   (custom: ~/.a3s/agents/*.yaml)
```

**How the three core concepts relate:**

| Concept | Responsibility |
|---------|---------------|
| `AgentDefinition` / `AgentRegistry` | Static agent definitions (YAML/Markdown) — describes prompts, permissions, and tool access |
| `AgentSlot` | Runtime declaration type — specifies which agent type to launch and its role |
| `AgentOrchestrator` | Execution engine — drives AgentLoop via `spawn` or `run_team` |

### Custom Subagents

Place YAML or Markdown files in your agent directories — they're loaded automatically at startup:

```hcl
# agent.hcl
agent_dirs = ["~/.a3s/agents", "/shared/team-agents"]
```

**YAML format:**

```yaml
# ~/.a3s/agents/security-reviewer.yaml
name: security-reviewer
description: Reviews code for security vulnerabilities and best practices
mode: subagent
hidden: false
max_steps: 25
can_spawn_subagents: false

permissions:
  allow:
    - read
    - grep
    - glob
    - ls
  deny:
    - write
    - edit
    - bash
    - patch

prompt: |
  You are a security code reviewer. Analyze the codebase for:
  - SQL injection, XSS, CSRF vulnerabilities
  - Hardcoded secrets and credentials
  - Insecure cryptographic practices
  Report findings with severity (Critical/High/Medium/Low),
  affected file and line, and recommended fix.
  Never modify any files — report only.
```

**Markdown format** (YAML frontmatter; body becomes the system prompt):

```markdown
---
name: db-migration
description: Database migration specialist
mode: subagent
max_steps: 40
permissions:
  allow: [read, write, edit, bash, grep, glob]
  deny:  [web_fetch, web_search]
---
You are a database migration specialist...
```

Use the `model` field to assign a different model to a specific agent:

```yaml

  allow: [read, grep, glob, ls]
```

## AgentSlot — Unified Declaration Type

`AgentSlot` is the single type for describing "how to launch an agent", used for both standalone subagents and team members:

```rust
pub struct AgentSlot {
    pub agent_type: String,             // "general" | "explore" | "plan" | custom name
    pub role: Option<TeamRole>,         // None = standalone; Some(Lead/Worker/Reviewer) = team member
    pub description: String,
    pub prompt: String,
    pub permissive: bool,               // true = skip human confirmation
    pub max_steps: Option<usize>,
    pub timeout_ms: Option<u64>,
    pub workspace: String,
    pub agent_dirs: Vec<String>,        // additional custom agent directories
    // ...
}
```

The `role` field determines the execution path:
- `role = None` → `spawn()` runs as a standalone subagent
- `role = Some(Lead)` / `Some(Worker)` / `Some(Reviewer)` → `run_team()` assembles a team

## Path 1: Standalone Subagent (spawn)

`spawn(AgentSlot)` runs a single agent in an isolated child session with real-time monitoring and dynamic control.

### Quick Start

<Tabs items={['Rust', 'TypeScript', 'Python']}>
<Tab value="Rust">

```rust
use a3s_code_core::{Agent, orchestrator::{AgentOrchestrator, AgentSlot}};
use std::sync::Arc;

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let agent = Arc::new(Agent::new("agent.hcl").await?);
    let orch = AgentOrchestrator::from_agent(agent);

    let slot = AgentSlot::new("general", "Find all Rust source files")
        .with_description("Find Rust files")
        .with_permissive(true)
        .with_max_steps(10);

    let handle = orch.spawn(slot).await?;
    let result = handle.wait().await?;
    println!("{result}");
    Ok(())
}
```

## Path 2: Team Workflow (run_team)

`run_team(goal, workspace, slots)` automatically assembles a Lead → Worker → Reviewer team, decomposes the goal into tasks, and coordinates execution.

<Callout type="info">
  **Prerequisite**: Requires `Orchestrator.create(agent)` with a real Agent attached.
</Callout>

### Quick Start

<Tabs items={['Rust', 'TypeScript', 'Python']}>
<Tab value="Rust">

```rust
use a3s_code_core::{Agent, orchestrator::{AgentOrchestrator, AgentSlot, TeamRole}};
use std::sync::Arc;

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let agent = Arc::new(Agent::new("agent.hcl").await?);
    let orch = AgentOrchestrator::from_agent(agent);

    let slots = vec![
        AgentSlot::new("general", "").with_role(TeamRole::Lead).with_max_steps(5),
        AgentSlot::new("general", "").with_role(TeamRole::Worker).with_max_steps(5),
        AgentSlot::new("general", "").with_role(TeamRole::Reviewer).with_max_steps(3),
    ];

    let result = orch.run_team(
        "List 3 common Rust data structures and briefly describe each",
        ".",
        slots,
    ).await?;

    println!("Done tasks:     {}", result.done_tasks.len());
    println!("Rejected tasks: {}", result.rejected_tasks.len());
    println!("Rounds:         {}", result.rounds);
    for task in &result.done_tasks {
        println!("  [{}] {}", task.id, task.result.as_deref().unwrap_or(""));
    }
    Ok(())
}
```

### Workflow Internals

```
1. Lead decomposes
   ├─ Receives goal, outputs JSON: {"tasks": ["task 1", "task 2", ...]}
   └─ Calls board.post() for each task

2. Workers execute (concurrent)
   ├─ Loop: claim() → session.send(task) → board.complete()
   └─ Sleep poll_interval_ms when no tasks available

3. Reviewer approves
   ├─ For each InReview task, evaluates the result
   ├─ "APPROVED: <reason>"   → board.approve() → task becomes Done
   └─ "REJECTED: <feedback>" → board.reject() → task re-queued for retry

4. Termination
   └─ All tasks done, or max_rounds reached
```

### TeamRole

| Role | Rust | TypeScript/Python | Responsibility |
|------|------|-------------------|---------------|
| Lead | `TeamRole::Lead` | `"lead"` | Decomposes goal, posts tasks |
| Worker | `TeamRole::Worker` | `"worker"` | Claims and executes tasks |
| Reviewer | `TeamRole::Reviewer` | `"reviewer"` | Approves or rejects results |

### TeamTaskBoard

The board is shared across all members via `Arc<TeamTaskBoard>`. Every operation is thread-safe:

```rust
let board = team.task_board();     // &TeamTaskBoard
let board = team.task_board_arc(); // Arc<TeamTaskBoard>

// Post a task — returns Some(task_id) or None if board is full
let id = board.post("Refactor JWT validation", "lead", None).unwrap();
// Pre-assign to a specific worker; task starts as InProgress (not Open)
let id = board.post("Add CSRF protection", "lead", Some("worker-1")).unwrap();

// Claim the next Open or Rejected task
let task: Option<TeamTask> = board.claim("worker-1");

// Complete a task (moves to InReview)
board.complete(&task.id, "Added RS256 key rotation");

// Reviewer actions
board.approve(&task.id);   // InReview → Done
board.reject(&task.id);    // InReview → Rejected (re-claimable by workers)

// Query
let tasks: Vec<TeamTask>    = board.by_status(TaskStatus::Done);
let tasks: Vec<TeamTask>    = board.by_assignee("worker-1");
let task:  Option<TeamTask> = board.get(&id);

// Stats — (open, in_progress, in_review, done, rejected)
let (open, prog, rev, done, rej) = board.stats();
let total = board.len();
```

### Task State Machine

```
post()      claim()       complete()
  Open ──► InProgress ──► InReview
                               │
                   ┌───────────┴──────────┐
              approve()              reject()
                   │                      │
                 Done           Rejected ──► (re-claim())
```

### TeamRunner

`TeamRunner` binds real `AgentSession` instances (or any `AgentExecutor`) to team members and runs the full automated workflow:

```rust
use a3s_code_core::agent_teams::TeamRunner;
use std::sync::Arc;

let mut runner = TeamRunner::new(team); // team is consumed here

runner.bind_session("lead",     Arc::new(agent.session(".", None)?))?;
runner.bind_session("worker-1", Arc::new(agent.session(".", None)?))?;
runner.bind_session("reviewer", Arc::new(agent.session(".", None)?))?;

let result = runner.run_until_done("Audit the auth module").await?;
```

`run_until_done` internals:

1. **Lead decomposition** — the Lead's session receives a prompt to output JSON tasks. The runner parses the JSON and calls `board.post()` for each task.

2. **Worker execution** — one Tokio task per Worker. Each loops: `claim() → session.send() → board.complete()`

3. **Reviewer loop** — for each InReview task, the Reviewer's session evaluates the result and responds `APPROVED: <reason>` or `REJECTED: <feedback>`

4. **Termination** — when all tasks are Done, or after `max_rounds` Reviewer polling rounds

### AgentExecutor Trait

`TeamRunner` accepts any type implementing `AgentExecutor` — not just `AgentSession`. Useful for testing or custom execution strategies:

```rust
use a3s_code_core::agent_teams::AgentExecutor;

#[async_trait::async_trait]
impl AgentExecutor for MyCustomExecutor {
    async fn execute(&self, prompt: &str) -> a3s_code_core::Result<String> {
        // your LLM call or custom logic here
        Ok("result".to_string())
    }
}

runner.bind_session("worker-1", Arc::new(MyCustomExecutor::new()))?;
```

## LLM-Driven Delegation

Within a session, the LLM delegates work to subagents via the `task` and `parallel_task` tools. The built-in `delegate-task` skill (Instruction kind) is automatically injected into the system prompt, teaching the LLM when to delegate, which agents are available, and how to invoke the tools — no additional configuration required.

### How Delegation Works

```
Parent Session
  │
  ├─ task("security-reviewer", "Review auth module")
  │    │
  │    ├─ 1. Look up AgentDefinition in AgentRegistry
  │    ├─ 2. Create child session with agent's permissions
  │    ├─ 3. Apply agent's system prompt
  │    ├─ 4. Execute task in isolated agentic loop
  │    ├─ 5. Return result to parent
  │    └─ 6. Child session ID preserved for reference
  │
  └─ Parent continues with subagent's result
```

Key isolation guarantees:
- Child session has its own context window (no parent context leakage)
- Permissions are restricted to the agent definition's policy
- `max_steps` prevents runaway execution
- `can_spawn_subagents: false` prevents recursive delegation

### Subagent Events

When streaming, subagent lifecycle events are emitted on the parent session's event channel:

```rust
let (mut rx, _handle) = session.stream("Analyze this codebase thoroughly").await?;
while let Some(event) = rx.recv().await {
    match event {
        AgentEvent::SubagentStart { task_id, agent, description, .. } => {
            println!("▶ [{agent}] {description} (task: {task_id})");
        }
        AgentEvent::SubagentEnd { agent, success, output, .. } => {
            let status = if success { "✓" } else { "✗" };
            println!("{status} [{agent}] {}", output.chars().take(100).collect::<String>());
        }
        AgentEvent::End { .. } => break,
        _ => {}
    }
}
```

## Multi-Session Parallelism

For workloads that go beyond subagent delegation — e.g., processing multiple repositories or running the same analysis across different workspaces — create multiple independent sessions and drive them concurrently using language-native primitives.

<Tabs items={['Rust', 'TypeScript', 'Python']}>
<Tab value="Rust">

```rust
use a3s_code_core::Agent;
use tokio::task::JoinSet;

let agent = Agent::new("agent.hcl").await?;

let workspaces = vec![
    "/repos/service-auth",
    "/repos/service-billing",
    "/repos/service-gateway",
];

// Spawn one session per workspace, all running concurrently
let mut join_set = JoinSet::new();
for workspace in workspaces {
    let agent = agent.clone();
    join_set.spawn(async move {
        let session = agent.session(workspace, None)?;
        let result = session.send("Find all TODO comments and summarize them").await?;
        Ok::<_, a3s_code_core::CodeError>((workspace.to_string(), result.text))
    });
}

// Collect results as they complete
while let Some(result) = join_set.join_next().await {
    match result? {
        Ok((workspace, text)) => println!("=== {workspace} ===\n{text}\n"),
        Err(e) => eprintln!("Error: {e}"),
    }
}
```

### Fan-Out / Fan-In Pattern

Combine multi-session parallelism with subagent delegation for fan-out/fan-in workflows:

```rust
use a3s_code_core::Agent;
use tokio::task::JoinSet;

let agent = Agent::new("agent.hcl").await?;

// Phase 1: Parallel exploration across workspaces
let mut join_set = JoinSet::new();
for workspace in &["/repos/frontend", "/repos/backend", "/repos/infra"] {
    let agent = agent.clone();
    let ws = workspace.to_string();
    join_set.spawn(async move {
        let session = agent.session(&ws, None)?;
        // Each session uses parallel_task internally to fan out further
        let result = session.tool("parallel_task", serde_json::json!({
            "tasks": [
                { "agent": "explore",           "description": "Find dependencies", "prompt": "List all external dependencies." },
                { "agent": "security-reviewer", "description": "Security scan",     "prompt": "Scan for known vulnerability patterns." }
            ]
        })).await?;
        Ok::<_, a3s_code_core::CodeError>((ws, result.output))
    });
}

// Phase 2: Collect findings
let mut findings = Vec::new();
while let Some(result) = join_set.join_next().await {
    if let Ok(Ok((ws, output))) = result {
        findings.push(format!("## {ws}\n{output}"));
    }
}

// Phase 3: Summarize into a single session
let summary_session = agent.session("/repos", None)?;
let summary = summary_session.send(&format!(
    "Summarize these findings into a single security report:\n\n{}",
    findings.join("\n\n")
)).await?;

println!("{}", summary.text);
```

## Choosing a Mode

| Scenario | Recommended |
|----------|-------------|
| LLM autonomously decides whether to delegate | LLM delegation (task / parallel_task) |
| Launch a single specialized task from code | `spawn(AgentSlot)` |
| Need real-time monitoring, pause/resume control | `spawn(AgentSlot)` |
| Goal needs decomposing into parallel subtasks | `run_team(goal, ws, slots)` |
| Need Lead/Worker/Reviewer collaboration and quality review | `run_team(goal, ws, slots)` |
| Need full control over task board and orchestration logic | `AgentTeam` + `TeamRunner` directly |
| Multi-repo or cross-workspace parallel analysis | Multi-session parallelism (JoinSet / Promise.all) |
