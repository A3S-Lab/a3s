# Agentic Search - Technical Deep Dive

This guide explains the technical principles behind the `agentic_search` tool and provides practical implementation examples.

<Callout type="info">
**Inspired by Sirchmunk**: The agentic_search tool implements a multi-phase search pipeline inspired by the [Sirchmunk paper](https://github.com/modelscope/sirchmunk), replacing traditional embedding-based vector search with intelligent keyword extraction and relevance ranking.
</Callout>

---

## Technical Principles

### Architecture Overview

```
┌─────────────────────────────────────────────────────────────┐
│ User Query: "how does JWT token validation work"           │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 1: Keyword Extraction                                 │
│ - Remove stop words ("how", "does", "work")               │
│ - Extract phrases: ["JWT token", "token validation"]      │
│ - Extract words: ["jwt", "token", "validation"]           │
│ Result: ["JWT token", "token validation", "jwt", ...]     │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 2: Parallel Search                                    │
│ ├─ Content Search: ripgrep-style regex matching           │
│ │  - Search file contents for keyword patterns            │
│ │  - Collect matching lines with context                  │
│ └─ Structure Search: filename and path matching           │
│    - Match keywords against file names                     │
│    - Match keywords against directory paths                │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 3: Relevance Scoring                                  │
│ - Base score: match_count / sqrt(file_lines)              │
│ - IDF boost: unique_keywords_matched / total_keywords     │
│ - File type boost: code(1.2x) > config(1.0x) > docs(0.9x)│
│ - Final: base_score * idf_boost * type_boost              │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 4: Result Formatting                                  │
│ - Sort by relevance descending                             │
│ - Format with context lines                                │
│ - Return top N results                                      │
└─────────────────────────────────────────────────────────────┘
```

### Phase 1: Keyword Extraction

The keyword extraction algorithm removes stop words and extracts meaningful search terms at multiple granularities:

**Algorithm:**
```rust
fn extract_keywords(query: &str) -> Vec<String> {
    let mut keywords = Vec::new();

    // 1. Extract 2-3 word phrases
    let words: Vec<&str> = query.split_whitespace().collect();
    for window in words.windows(2) {
        keywords.push(window.join(" "));
    }

    // 2. Extract individual meaningful words (length >= 3)
    for word in &words {
        let clean = word.trim_matches(|c: char| !c.is_alphanumeric())
                        .to_lowercase();
        if clean.len() >= 3 && !STOP_WORDS.contains(&clean.as_str()) {
            keywords.push(clean);
        }
    }

    // 3. Deduplicate and limit to top 8
    keywords.dedup();
    keywords.truncate(8);
    keywords
}
```

**Example:**
```
Input:  "how does JWT token validation work"
Output: ["JWT token", "token validation", "jwt", "token", "validation"]
```

### Phase 2: IDF-Weighted Relevance Scoring

Files matching more unique keywords rank higher. This is inspired by TF-IDF (Term Frequency-Inverse Document Frequency) but simplified for code search.

**Scoring Formula:**
```rust
// Base score: normalized match density
let base_score = (match_count as f32) / (total_lines as f32).sqrt();

// IDF boost: keyword coverage
let unique_keywords_matched = patterns
    .iter()
    .filter(|p| lines.iter().any(|l| p.is_match(l)))
    .count();
let idf_boost = (unique_keywords_matched as f32) / (patterns.len() as f32);

// File type boost
let type_boost = match file_type {
    Code => 1.2,           // Prioritize code files
    Config => 1.0,         // Standard weight
    Documentation => 0.9,  // Slightly lower
    Other => 0.8,          // Lowest priority
};

// Final relevance
let relevance = base_score * idf_boost * type_boost;
```

**Why this works:**
- **Base score**: Smaller, focused files rank higher than large files with scattered matches
- **IDF boost**: Files matching more keywords are more relevant
- **Type boost**: Code files are prioritized over documentation

### Phase 3: Monte Carlo Evidence Sampling (DEEP Mode)

DEEP mode uses Gaussian importance sampling to explore code regions around high-scoring matches.

**Three-Act Strategy:**

**Act 1: Fuzzy Anchoring**
```rust
// Identify high-scoring regions as anchors
let anchors: Vec<usize> = file_matches
    .iter()
    .map(|m| m.line_number)
    .collect();
```

**Act 2: Gaussian Importance Sampling**
```rust
// Sample around each anchor with Gaussian weights
let sigma = (5.0 - match_density * 20.0).max(2.0);  // Adaptive sigma

for &anchor in &anchors {
    let window_radius = (sigma * 3.0) as usize;
    for line_idx in (anchor - window_radius)..(anchor + window_radius) {
        let distance = (line_idx as i32 - anchor as i32).abs();

        // Gaussian weight: exp(-d² / (2σ²))
        let weight = (-((distance * distance) as f32) / (2.0 * sigma * sigma)).exp();

        sampled.push(SampledLine { line_idx, weight, ... });
    }
}
```

**Act 3: Evidence Synthesis**
```rust
// Compute composite evidence score
let weight_sum: f32 = sampled.iter().map(|s| s.weight).sum();
let evidence_score = file_relevance * weight_sum / (sampled.len() as f32).sqrt();
```

**Why Gaussian sampling?**
- Focuses on regions around matches (not random sampling)
- Decaying weights ensure nearby lines are prioritized
- Adaptive sigma adjusts to match density

## Performance Characteristics

### Benchmark Results

On a typical mid-size project (~50k lines):

| Operation | Time | Notes |
|-----------|------|-------|
| Single keyword (FAST) | ~100ms | "authentication" |
| Multi-keyword (FAST) | ~150ms | "JWT token validation" |
| DEEP mode | ~15s | Monte Carlo sampling |
| Filename search | ~50ms | filename_only mode |
| Large codebase (500k lines) | ~800ms | With include patterns |

### Optimization Tips

1. **Use include patterns** to limit file types:
   ```
   include: "*.{rs,toml}"  // Only Rust and TOML files
   ```

2. **Adjust max_results** based on needs:
   ```
   max_results: 3   // Quick overview
   max_results: 10  // Standard (default)
   max_results: 20  // Comprehensive
   ```

3. **Use FILENAME_ONLY for discovery**:
   ```
   mode: "filename_only"  // 10x faster than content search
   ```

4. **Reserve DEEP mode for complex queries**:
   ```
   mode: "deep"  // Only when FAST mode is insufficient
   ```

## Troubleshooting

### Search is Too Slow

**Problem:** Search takes > 5 seconds

**Solutions:**
1. Use `include` pattern to limit file types
2. Reduce `max_results`
3. Use `filename_only` mode for quick discovery

### Expected Results Not Found

**Problem:** Relevant files are missing from results

**Solutions:**
1. Try more specific keywords
2. Check if files are excluded by `.gitignore`
3. Verify file size is < 1MB (large files are skipped)
4. Try `grep` for exact matching

### Too Many Results

**Problem:** Results are overwhelming

**Solutions:**
1. Make query more specific
2. Use `include` pattern to filter file types
3. Reduce `max_results`
4. Reduce `context_lines`
