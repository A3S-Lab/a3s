# Agent Harness Protocol (AHP)

The **Agent Harness Protocol** is a language-agnostic protocol for supervising A3S Code agent sessions from an external process. Where [lifecycle hooks](/docs/en/code/hooks) let you intercept events in-process (Rust, Python, TypeScript), AHP lets you do the same thing from **any language** — by spawning a child process and communicating via newline-delimited JSON-RPC 2.0 over stdio.

AHP is modeled on the same transport principle as MCP: the A3S Code host spawns a harness server process, sends events over stdin, and reads decisions from stdout. The harness server can be written in Go, Java, a shell script, or any other language.

## Architecture

```
A3S Code host
  └── AgentSession
        └── AhpHookExecutor  (implements HookExecutor)
              ├── pre_tool_use / pre_prompt  →  JSON-RPC request  →  awaits response
              └── all other events           →  JSON-RPC notification  →  no wait

Harness server (child process, any language)
  ├── reads newline-delimited JSON from stdin
  └── writes JSON-RPC responses to stdout
```

The harness server's stderr is inherited by the host, so all server-side log output appears in the host terminal.

## Protocol

### Transport

Events are sent as newline-delimited JSON over the child process's stdin/stdout. Each message is a single JSON object terminated by a newline character (`\n`).

### Blocking vs. fire-and-forget

<TypeTable
  type={{
    "pre_tool_use": {
      description: "Blocking — host sends a request with an id and waits for a response before the tool executes",
    },
    "pre_prompt": {
      description: "Blocking — host waits for a response before the prompt is sent to the LLM",
    },
    "post_tool_use, session_start, session_end, generate_start, generate_end, post_response, on_error, skill_load, skill_unload": {
      description: "Fire-and-forget — host sends a notification with no id and does not wait",
    },

/>

### Request (blocking event)

The host sends this when a decision is required before execution continues:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "harness/event",
  "params": {
    "event_type": "pre_tool_use",
    "payload": {
      "session_id": "s1",
      "tool": "Bash",
      "args": { "command": "ls -la" },
      "working_directory": "/workspace",
      "recent_tools": []
    },
    "meta": { "depth": 0 }
  }
}
```

The harness server must respond with a JSON-RPC result before the default timeout (10 seconds) elapses:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "action": "continue",
    "reason": null,
    "modified": null,
    "retry_delay_ms": null
  }
}
```

### Notification (fire-and-forget event)

The host sends this for observability events that require no decision:

```json
{
  "jsonrpc": "2.0",
  "method": "harness/event",
  "params": {
    "event_type": "post_tool_use",
    "payload": { "session_id": "s1", "tool": "Bash", ... },
    "meta": { "depth": 0 }
  }
}
```

No response is expected or sent for notifications.

### `meta.depth` — sub-agent nesting

Every AHP message carries a `meta` object with a `depth` field:

<TypeTable
  type={{
    "depth: 0": {
      description: "User-facing session (top-level agent)",
    },
    "depth: 1": {
      description: "First-level sub-agent spawned by the task tool",
    },
    "depth: N": {
      description: "N-th level sub-agent. The harness server can apply stricter policies to deeper agents.",
    },
  }}
/>

### Decision actions

<TypeTable
  type={{
    "continue": {
      description: "Allow the operation to proceed. If `modified` is set, replace the event payload before execution.",
    },
    "block": {
      description: "Cancel the operation. The `reason` string is returned to the agent as an error.",
    },
    "skip": {
      description: "Skip the operation silently (no error returned to the agent).",

If the harness server does not respond within the timeout, the host falls back to `continue` so the agent is never permanently stalled.

## Writing a harness server

A harness server is any process that reads newline-delimited JSON from stdin and writes JSON-RPC responses to stdout. All logging must go to stderr so it does not interfere with the protocol.

### Python reference server

The repository ships a reference Python server at `core/examples/ahp_server.py`. It demonstrates:

- Parsing the `event_type` and `payload` from each message
- Reading `meta.depth` for depth-aware decisions
- Blocking dangerous Bash commands via regex patterns
- Detecting sensitive output patterns in `post_tool_use` (fire-and-forget)
- Writing all log output to stderr

Key structure:

```python
import json, sys

def on_pre_tool_use(payload, depth):
    tool = payload.get("tool", "")
    command = payload.get("args", {}).get("command", "")
    # depth-aware: stricter policy for sub-agents
    if depth > 0 and tool == "Bash" and is_dangerous(command):
        return {"action": "block", "reason": f"blocked at depth {depth}"}
    return {"action": "continue"}

for raw_line in sys.stdin:
    msg = json.loads(raw_line.strip())
    params = msg.get("params", {})
    event_type = params.get("event_type", "")
    payload    = params.get("payload", {})
    depth      = params.get("meta", {}).get("depth", 0)
    req_id     = msg.get("id")          # None for notifications

    if req_id is None:
        dispatch_notification(event_type, payload, depth)
        continue

    result = dispatch_request(event_type, payload, depth) or {"action": "continue"}
    print(json.dumps({"jsonrpc": "2.0", "id": req_id, "result": result}), flush=True)
```

### Minimal shell harness

A shell script that allows everything (useful as a starting point):

```bash
#!/usr/bin/env bash
# harness.sh — allow-all AHP server
while IFS= read -r line; do
    id=$(echo "$line" | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('id',''))")
    if [ -n "$id" ]; then
        echo "{\"jsonrpc\":\"2.0\",\"id\":$id,\"result\":{\"action\":\"continue\"}}"
    fi
done
```

## Sub-agent depth propagation

When a session with an attached harness spawns sub-agents via the `task` tool, the `AhpHookExecutor` is propagated automatically — the same harness server process receives events from all sub-agents. The `meta.depth` field increments with each nesting level, letting the server apply stricter policies at deeper depths.

```
Parent session (depth: 0)
  └── task tool spawns Child (depth: 1)
        └── task tool spawns Grandchild (depth: 2)
              └── ...
```

All events from all levels are multiplexed into the same stdin/stdout stream. The harness server can distinguish origins using `session_id` (unique per agent) and `depth`.

<Callout type="info">
  AHP depth propagation uses the `ToolContext.hook_executor` mechanism — `AhpHookExecutor` is registered there and inherited at every nesting level.
</Callout>
