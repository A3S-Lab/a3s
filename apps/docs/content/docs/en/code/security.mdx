# Security

A3S Code provides two built-in security layers: **Permission Policy** and **Human-in-the-Loop (HITL) Confirmation**. Additional security logic can be implemented via the `SecurityProvider` trait and `HookEngine`.

## Security Architecture

```
User Prompt → Agent Loop
  ├─ Permission Policy    ← Deny → Allow → Ask → Default
  ├─ HITL Confirmation    ← Independent of permissions
  ├─ Tool Execution
  └─ SecurityProvider     ← Pluggable via trait (taint tracking, sanitization, etc.)
```

## Permission Policy

The permission system controls which tools the agent can use. Rules are evaluated in order: **Deny → Allow → Ask → Default**.

### Pattern Matching

Rules use glob-style pattern matching on tool names and arguments:

```
Tool(pattern)
```

<TypeTable
  type={{
    "bash(rm -rf:*)": {
      description: "Any bash command starting with rm -rf",
    },
    "read(src/**)": {
      description: "Read operations on any file under src/",
    },
    "write(*)": {
      description: "All write operations",
    },
    "mcp__pencil": {
      description: "All mcp__pencil__* tools (prefix matching)",
    },
  }}
/>

Wildcards: `*` matches any character except `/`, `**` matches including `/`, `:*` matches any suffix.

### Building a Custom Policy

<Tabs groupId="lang" items={['TypeScript', 'Python']}>
<Tab value="TypeScript">
```typescript
import { PermissionPolicy } from '@a3s-lab/code';

const policy = new PermissionPolicy()
  .deny('bash(rm -rf:*)')
  .deny('bash(curl:*|sh)')
  .allow('read(*)')
  .allow('glob(*)')
  .allow('grep(*)')
  .ask('write(*)')
  .ask('bash(*)');

await session.setPermissionPolicy(policy);
```
</Tab>
<Tab value="Python">

### Evaluation Order

Rules are evaluated in this order:

<Steps>
<Step>**Deny rules** — If any deny rule matches, block immediately</Step>
<Step>**Allow rules** — If any allow rule matches, permit</Step>
<Step>**Ask rules** — If any ask rule matches, trigger HITL confirmation</Step>
<Step>**Default policy** — If no rules match, use the default (permissive or strict)</Step>
</Steps>

## Human-in-the-Loop (HITL)

HITL confirmation is **independent** of the permission system. Even if a tool is `allow`ed by permissions, HITL can still require user confirmation.

### Configuration

<Tabs groupId="lang" items={['TypeScript', 'Python']}>
<Tab value="TypeScript">
```typescript
const session = agent.session('/project', {
  confirmationPolicy: {
    enabled: true,
    timeoutMs: 30000,
    timeoutAction: 'reject',
  },
});
```
</Tab>
<Tab value="Python">
```python
from a3s_code import ConfirmationPolicy

session = agent.session("/project",
    confirmation_policy=ConfirmationPolicy(
        enabled=True,
        timeout_ms=30000,
        timeout_action="reject",
    ))
```
</Tab>
</Tabs>

### Handling Confirmation Events

<Tabs groupId="lang" items={['TypeScript', 'Python']}>
<Tab value="TypeScript">
```typescript
for await (const event of session.stream('Clean up old logs')) {
  if (event.type === 'confirmation_required') {
    console.log(`🔒 Approve '${event.toolName}'?`, event.toolArgs);
    await session.respondConfirmation(event.toolId, { approved: true });
  } else if (event.type === 'permission_denied') {
    console.log(`🚫 ${event.toolName} blocked: ${event.reason}`);
  } else if (event.type === 'text_delta') {
    process.stdout.write(event.text);
  }
}
```
</Tab>
<Tab value="Python">
```python
for event in session.stream("Clean up old logs"):
    if event.event_type == "confirmation_required":
        print(f"🔒 Approve '{event.tool_name}'? {event.tool_args}")
        session.respond_confirmation(event.tool_id, approved=True)
    elif event.event_type == "permission_denied":
        print(f"🚫 {event.tool_name} blocked: {event.reason}")
    elif event.event_type == "text_delta":
        print(event.text, end="", flush=True)
```
</Tab>
</Tabs>

## Workspace Boundaries

All file operations are restricted to the session's workspace directory. Attempts to access files outside the workspace are blocked.

## Built-in Security Provider

A3S Code ships a `DefaultSecurityProvider` that is off by default. Call `with_default_security()` to activate it — no additional configuration required.

### Enabling

<Tabs groupId="lang" items={['TypeScript', 'Python']}>
<Tab value="TypeScript">
```typescript
import { DefaultSecurityProvider } from '@a3s-lab/code';

// Default security config
const session = agent.session('/project', {
  securityProvider: new DefaultSecurityProvider(),
});

// Custom config
const session = agent.session('/project', {
  securityConfig: {
    enableTaintTracking: true,
    enableOutputSanitization: true,
    enableInjectionDetection: true,
    customPatterns: [
      { name: 'internal_id', pattern: 'INT-\\d{6}', label: 'REDACTED:ID' },
    ],
  },
});
```
</Tab>
<Tab value="Python">
```python
from a3s_code import DefaultSecurityProvider

# Default security config
session = agent.session("/project",
```

### Prompt Injection Detection

`DefaultSecurityProvider` scans user input in the `GenerateStart` hook for injection patterns:

| Pattern | Example |
|---------|---------|
| Ignore prior instructions | "Ignore all previous instructions..." |
| Disregard context | "Disregard all prior context..." |
| Mode switch | "You are now in developer mode..." |
| Forget instructions | "Forget everything you learned..." |
| New instruction injection | "New instructions: ..." |
| System prompt override | "System prompt override" |

When injection is detected, it is reported via the `OnError` hook and emitted as a `SecurityAlert` in the agent event stream. Taint tracking also scans tool outputs to catch indirect injection (e.g., via poisoned files or web content read by the agent).

## Best Practices

<Steps>
<Step>**Start with strict policy** — Use `PermissionPolicy::strict()` and explicitly allow only needed tools</Step>
<Step>**Block dangerous patterns** — Always deny destructive bash commands: `bash(rm -rf:*)`, `bash(curl:*|sh)`</Step>
<Step>**Enable HITL for writes** — Require confirmation for file modifications and bash execution</Step>
<Step>**Use workspace boundaries** — Never disable workspace restrictions</Step>
<Step>**Implement SecurityProvider** — Add custom taint tracking and output sanitization for sensitive data</Step>
<Step>**Audit logs** — Use hooks to log all tool executions for security review</Step>
</Steps>
