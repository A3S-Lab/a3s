# Sessions

Each session is bound to a workspace directory. The Agent creates sessions via `agent.session(workspace, options?)`. Sessions hold their own LLM client, conversation history, and tool context.

The generation APIs — `send()` and `stream()` — send prompts to the LLM and return responses. The agent loop handles tool execution automatically.

## Create Session

<Tab value="TypeScript">

```typescript
  model: 'openai/gpt-4o',
```

## System Prompt Customization (Slot-Based)

Customize the agent's behavior without overriding core agentic capabilities. Four named slots let you inject custom personality while preserving tool usage strategy, autonomous behavior, and completion criteria:

| Slot | Position | Behavior |
|------|----------|----------|
| `role` | Before core | Replaces default "You are A3S Code..." identity |
| `guidelines` | After core | Appended as `## Guidelines` section |
| `response_style` | Replaces section | Replaces default `## Response Format` |
| `extra` | End | Freeform instructions (backward-compatible) |

<Tabs groupId="lang" items={['TypeScript', 'Python']}>
<Tab value="TypeScript">
```typescript
const session = agent.session('/my-project', {
  role: 'You are a senior Rust developer.',
  guidelines: 'Use clippy. No unwrap(). Prefer Result.',
  responseStyle: 'Be concise. Use bullet points.',
  extra: 'This project uses tokio and axum.',
});
```
</Tab>
<Tab value="Python">
```python
from a3s_code import SessionOptions

opts = SessionOptions()

opts.guidelines = "Use clippy. No unwrap(). Prefer Result."
```

## Error Recovery & Resilience

Three layers of error recovery protect long-running sessions:

<Accordions>
<Accordion title="Parse Error Recovery">
When the LLM returns malformed tool arguments (`__parse_error`), the error is fed back as a tool result so the model can self-correct. After `max_parse_retries` consecutive failures the loop aborts.
</Accordion>
<Accordion title="Tool Execution Timeout">
Each tool call is wrapped in `tokio::time::timeout`. A timed-out tool produces an error message that is fed back to the LLM; the session continues.
</Accordion>
<Accordion title="Circuit Breaker">
Transient LLM API failures (network errors, rate limits) are retried up to `circuit_breaker_threshold` times in non-streaming mode with short exponential backoff. In streaming mode any failure is fatal (events cannot be replayed).
</Accordion>
</Accordions>

<Tabs groupId="lang" items={['TypeScript', 'Python']}>
<Tab value="TypeScript">
```typescript
// Individual controls
const session = agent.session('.', {
  maxParseRetries: 3,        // bail after 3 consecutive parse errors
  toolTimeoutMs: 30000,      // 30s per tool
  circuitBreakerThreshold: 5 // retry LLM up to 5 times
});
```
</Tab>
<Tab value="Python">
```python

  max_parse_retries=3,        # bail after 3 consecutive parse errors
  tool_timeout_ms=30000,      # 30s per tool

)
```

## Stream

<Tabs groupId="lang" items={['TypeScript', 'Python']}>
<Tab value="TypeScript">
```typescript
// Returns an EventStream — use for await...of or call .next() manually
const stream = await session.stream('Refactor the auth module');
for await (const event of stream) {
  if (event.type === 'text_delta') process.stdout.write(event.text);
  if (event.type === 'tool_start') console.log(`\n🔧 ${event.toolName}`);
  if (event.type === 'tool_end') console.log(`  → ${event.toolOutput?.slice(0, 100)}`);
}

// Or iterate manually with .next()
const stream2 = await session.stream('Explain src/main.rs');
while (true) {
  const { value, done } = await stream2.next();
  if (done) break;
  if (value.type === 'text_delta') process.stdout.write(value.text);
}
```
</Tab>
<Tab value="Python">
```python
# Sync iteration (works without an event loop)
for event in session.stream("Refactor the auth module"):
    if event.event_type == "text_delta":
        print(event.text, end="", flush=True)
    elif event.event_type == "tool_start":
        print(f"\n🔧 {event.tool_name}")
    elif event.event_type == "tool_end":
        print(f"  → {event.tool_output[:100]}")

# Async iteration (inside async def)
async for event in session.stream("Refactor the auth module"):
    if event.event_type == "text_delta":
        print(event.text, end="", flush=True)
    elif event.event_type == "end":
        print(f"\nDone — {event.total_tokens} tokens")
        break
```

## Slash Commands

Session commands are intercepted before the LLM. Type `/help` in any session to list all available commands.

| Command | Description |
|---------|-------------|
| `/help` | List available commands |
| `/model [provider/model]` | Show or switch the current model |
| `/cost` | Show token usage and estimated cost |
| `/history` | Show conversation turn count and token stats |
| `/clear` | Clear conversation history |
| `/compact` | Manually trigger context compaction |
| `/tools` | List registered tools |
| `/mcp` | List connected MCP servers and their tools |
| `/loop [interval] <prompt>` | Schedule a recurring prompt (default interval: 10m) |
| `/cron-list` | List all scheduled recurring prompts |
| `/cron-cancel <id>` | Cancel a scheduled task by ID |

### Custom Commands

Register custom slash commands handled before the LLM:

<Tabs groupId="lang" items={['TypeScript', 'Python']}>
<Tab value="TypeScript">
```typescript
function statusHandler(args: string, ctx: CommandContext): string {
  console.log(`Model: ${ctx.model}, History: ${ctx.historyLen} msgs, args: ${args}`);
  return `Status: OK`;  // Note: return value is logged but not captured in current implementation
}

session.registerCommand("status", "Show session status", statusHandler);

const result = await session.send("/status hello");
console.log(result.text);
// Command 'status' executed successfully
```

The handler receives:
- `args` — string with everything after the command name
- `ctx` — object with `sessionId`, `model`, `historyLen`, `totalTokens`, `totalCost`, `toolNames`, and `mcpServers`

**Note**: Due to napi-rs limitations, the return value from the handler function is not currently captured. The command will execute successfully and you can use `console.log()` or other side effects in your handler. Future versions will support return value capture.

</Tab>

## Scheduled Tasks

Sessions support recurring prompts via the `/loop` slash command or the programmatic scheduler API. Scheduled tasks fire after each `send()` call when their interval has elapsed. The background ticker starts lazily on the first `send()` and stops automatically when the session is dropped.

### Schedule a Recurring Prompt

<Tabs groupId="lang" items={['TypeScript', 'Python']}>
<Tab value="TypeScript">
```typescript
// Via /loop slash command
const r = await session.send('/loop 30s check deployment status');
console.log(r.text);
// Scheduled [a1b2c3d4]: "check deployment status" — fires every 30s

// Programmatic (interval in seconds)
const taskId = session.scheduleTask('summarize recent commits', 300);  // every 5 min
console.log(`Scheduled: ${taskId}`);
```
</Tab>
<Tab value="Python">
```python
# Via /loop slash command
r = session.send('/loop 30s check deployment status')
print(r.text)
# Scheduled [a1b2c3d4]: "check deployment status" — fires every 30s

# Programmatic (interval in seconds)
task_id = session.schedule_task('summarize recent commits', 300)  # every 5 min
print(f'Scheduled: {task_id}')
```
</Tab>
</Tabs>

**Interval syntax:**

| Format | Example | Meaning |
|--------|---------|---------|
| Seconds | `30s` | Every 30 seconds |
| Minutes | `5m` | Every 5 minutes |
| Hours | `2h` | Every 2 hours |
| Days | `1d` | Every day |
| Leading | `/loop 5m <prompt>` | Interval before the prompt |
| Trailing | `/loop <prompt> every 5m` | `every` clause at end |
| Default | `/loop <prompt>` | Every 10 minutes |

<Callout type="info">
**When tasks fire:** Scheduled tasks are checked and executed after each `send()` call returns. A 1-second background ticker advances the internal clock; tasks whose interval has elapsed are queued and processed at the next `send()`. A reentrance guard (`is_processing_cron`) prevents nested firing.
